# 🧪 テストユーザー1名による実地検証手順書

本手順書は、SharePoint 永続化の実装が完了した React 版支援記録を、実際の運用環境で安全に検証するためのステップバイステップガイドです。

---

## 📅 1. 検証前準備

検証を開始する前に、以下の項目を準備します。

- [ ] **検証対象ユーザーの選定**: 
  - 1日19行の標準的なスケジュールを持つ利用者1名を特定する。
- [ ] **検証日の決定**: 
  - 本日、または特定の日付（過去日でも可、ただし他者が入力していないこと）。
- [ ] **リストアクセスの確認**: 
  - 検証実施者が `SupportRecord_Daily` および `SupportRecord_DailyRows` に直接アクセス（ブラウザで閲覧）できる状態にする。
- [ ] **Power Apps 側の一時利用停止連絡**: 
  - 検証中の特定ユーザー・特定日付に関しては、Power Apps から保存を行わないよう周知する（または実施者が独占して検証する）。

---

## 📝 2. 実機入力・保存手順（React）

React 画面から実際にデータを入力し、保存の挙動を確認します。

1.  **利用者・日付の選択**:
    - React 側の利用者選択画面から対象ユーザーを選び、検証対象日を指定する。
2.  **スロット入力（最初の数行）**:
    - 最初の1〜2行に対して、ステータス（完了/発動/スキップ）を選択し、メモを入力する。
3.  **オートアドバンスの確認**:
    - 保存後、自動的に次の未記入行へフォーカスが移動することを確認する。
4.  **19行完遂テスト**:
    - 可能な限りすべてのスロットを入力し、全行が期待通り保存されるか試行する。
5.  **日報コピー実行**:
    - 「日報をコピー」ボタンを押し、クリップボードの内容が入力内容と一致しているかペーストして確認する。

---

## 🔍 3. SharePoint 側の実データ確認ポイント

保存されたデータが SharePoint リスト上で意図した構造になっているか目視確認します。

- [ ] **親レコード (`SupportRecord_Daily`)**:
  - `Title` が `${date}-${userId}` になっているか。
  - `RecordDate` が正しく格納されているか。
- [ ] **子レコード (`SupportRecord_DailyRows`)**:
  - 対応する19行（または入力分）のアイテムが作成されているか。
  - `RowKey` が `${date}-${userId}-${slotKey}` の形式で一意に振られているか。
  - `ParentIDLookup` が、上記の親レコードの ID と一致しているか。
- [ ] **更新の挙動**:
  - React 側で一度保存した行を修正して再保存した際、SharePoint 側の行数が**増えずに値だけが上書き**されているか。

---

## ↩️ 4. 異常系の確認とロールバック手順

- [ ] **再読込テスト**:
  - ブラウザをリロードし、入力した内容が SharePoint から正しくフェッチされ、画面に再現されるか。
- [ ] **ロールバック（データ削除）**:
  - 検証終了後、本番データとして残したくない場合は、SharePoint リストから作成した親・子アイテムを手動で削除する。
  - ※ 削除した場合、React 画面をリロードすると「未記入」状態に戻ることを確認する。

---

## ✅ 5. 本番移行可否の判定基準（合格基準）

以下のすべてを満たした場合、検証合格と判断し、次のフェーズ（全ユーザー展開）へ進みます。

1. **データ不整合ゼロ**: 親子紐付けが100%正確である。
2. **パフォーマンス維持**: ページ読み込みや保存時に、体感的な遅延やタイムアウトが発生しない。
3. **5000件耐性**: `RowKey` によるフィルタリングが正常に機能している。
4. **UX維持**: オートアドバンスや日報コピーがストレスなく動作する。
5. **運用ルールの確立**: Power Apps との競合回避ルールが運用者に理解されている。
