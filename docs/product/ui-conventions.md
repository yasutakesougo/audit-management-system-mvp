## 1. Situation UI 原則

> **対応原則:** [原則 1 — 主役は AI ではなく現場導線である](principles.md#原則-1--主役は-ai-ではなく現場導線である)

### 設計方針

| 方針 | 説明 |
|------|------|
| **表はシンプル** | 業務画面は記録・確認・実行に集中する |
| **裏は強い** | 分析・提案はバックグラウンドで常に動く |
| **割り込まない** | AI 提案は業務導線を中断しない |

### 情報の優先度レイヤー

```
Layer 0: 業務コンテキスト（今日やること、申し送り）
Layer 1: 実行導線（記録、チェック、入力）
Layer 2: 状況通知（アラート、注意候補）       ← ここまでが常時表示
Layer 3: 提案・分析（見直し候補、下書き候補）   ← 明示的な操作で展開
Layer 4: 根拠・履歴（provenance、audit trail） ← 詳細表示時のみ
```

### UI 配置ルール

| 画面カテゴリ | 主要要素 | 提案の表示方法 |
|-------------|---------|---------------|
| **Today / Handoff** | 朝会サマリー、本日の注意点 | バナー型（折りたたみ可） |
| **Daily** | 記録入力、ABC 記録 | インライン注釈 |
| **Planning** | 支援計画シート編集 | サイドパネル or ダイアログ |
| **Monitoring** | モニタリング結果確認 | Proposal Card（後述） |
| **Dashboard** | 横断表示、制度チェック | サマリーカード |

### 禁止パターン

- ❌ 分析画面をホーム画面にしない
- ❌ AI 提案を業務画面の最上部に配置しない
- ❌ 提案を閉じるのに 2 ステップ以上必要にしない
- ❌ 記録入力中にモーダルで提案を表示しない

## 2. Proposal Card 標準

> **対応原則:** [原則 3 — 提案は候補として出す](principles.md#原則-3--提案は命令ではなく候補として提示する)、[原則 4 — 根拠を付ける](principles.md#原則-4--提案には必ず根拠を添える)

### 基本構造

すべての提案は以下の共通構造で表示する。

```
┌──────────────────────────────────────────────┐
│  🏷️ [カテゴリ]                 [緊急度バッジ] │
│                                              │
│  ── タイトル ──                               │
│  午後帯の離席増加                              │
│                                              │
│  ── 根拠 ──                                   │
│  📅 対象期間: 2026-02-15 〜 2026-03-01        │
│  📊 参照記録: 23件                            │
│  📈 前月比: +35%                              │
│  📋 ルール: ABC 頻度ルール v2                  │
│                                              │
│  ── 提案内容 ──                               │
│  • 休憩導線の見直し                            │
│  • 環境刺激の調整                              │
│                                              │
│  [✅ 採用]  [❌ 却下]  [🕐 後で確認]           │
│                                              │
│  ▸ 出典レコードを表示 (展開)                   │
└──────────────────────────────────────────────┘
```

### カテゴリラベル

| カテゴリ | 色 | アイコン | ソース |
|---------|-----|---------|--------|
| `見直し候補` | `warning.main` | ⚠️ | Monitoring 評価 |
| `下書き候補` | `info.main` | 📝 | モニタリング下書き生成 |
| `注意候補` | `error.main` | 🔴 | ABC パターン変化検出 |
| `比較材料` | `secondary.main` | 📊 | 期間比較分析 |

### 緊急度バッジ

| レベル | 色 | 表示 | 基準 |
|--------|-----|------|------|
| `urgent` | `error` | `🔴 要確認` | 制度上の期限超過、安全リスク |
| `recommended` | `warning` | `🟡 推奨` | パターン変化検出、改善余地 |
| `suggested` | `info` | `🔵 参考` | 傾向情報、比較材料 |

### MUI 実装ガイドライン

```tsx
// Proposal Card の共通コンポーネント設計
interface ProposalCardProps {
  category: 'review' | 'draft' | 'alert' | 'comparison';
  urgency: 'urgent' | 'recommended' | 'suggested';
  title: string;
  evidence: {
    period: string;
    recordCount: number;
    comparison?: string;
    rule?: string;
  };
  proposals: string[];
  sourceRecordIds: string[];
  onAccept: () => void;
  onDismiss: (reason?: string) => void;
  onDefer: () => void;
}
```

### 実装原則

- 全画面で同一の `<ProposalCard>` コンポーネントを使用する
- カテゴリ・緊急度はバリアントとして表現する
- 根拠セクションは常に表示する（折りたたまない）
- 出典レコードリンクは展開式にする（Layer 4）

## 3. Evidence / Provenance 表示

> **対応原則:** [原則 4 — 根拠付与](principles.md#原則-4--提案には必ず根拠を添える)、[原則 6 — 改善接続](principles.md#原則-6--記録は保存で終わらせず改善に接続する)

### 根拠の 3 点セット

すべての提案には以下を必ず持たせる。

```
提案本体   →  何を提案しているか
根拠       →  なぜその提案が出たか
参照元     →  どのレコードに基づくか
```

### Provenance Badge 規約

既存の `ProvenanceBadge` コンポーネントの色分けを全画面で統一する。

| ソース | 色コード | バッジテキスト | 表示場所 |
|-------|---------|-------------|---------|
| アセスメント (ICF) | `#1f6feb` (🔵) | `[ICF]` | Planning Sheet フィールド |
| 特性アンケート | `#238636` (🟢) | `[特性]` | Planning Sheet フィールド |
| モニタリング | `#d29922` (🟠) | `[モニタ]` | Planning Sheet フィールド |
| 申し送り分析 | `#8957e5` (🟣) | `[申送]` | Proposal Card |
| ABC 記録分析 | `#da3633` (🔴) | `[ABC]` | Proposal Card |
| 手動入力 | `#8b949e` (⚪) | — | バッジ表示なし |

### 出典レコードリンク

出典レコードへのリンクは以下の形式で表示する。

```
📎 出典: ABC-0042 (2026-02-28)  [クリックで詳細へ]
📎 出典: PDCA-0015 (2026-02-25) [クリックで詳細へ]
```

- クリックで該当レコードの詳細画面に遷移する
- 同一タブ内でナビゲーションする（新規タブを開かない）
- `source_type + source_id + date` の 3 要素を必ず表示する

## 4. Accept / Dismiss UI

> **対応原則:** [原則 5 — 最終決定は人が行う](principles.md#原則-5--最終決定は必ず人が行う)

### 承認フローの標準 UI

```
AI 提案  →  人が確認  →  人が採用/却下  →  履歴保存
```

### 3 ボタン標準

すべての提案に対して、以下の 3 つの操作を統一する。

| ボタン | ラベル | アクション | 色 |
|--------|--------|-----------|-----|
| **採用** | `✅ 採用` | 提案内容を反映 + 採用履歴を記録 | `success` |
| **却下** | `❌ 却下` | 提案を非表示 + 却下理由を記録 | `error` (outlined) |
| **後で確認** | `🕐 後で` | 提案を保留状態にする | `default` (text) |

### 却下理由の収集

却下時は、以下の選択肢から理由を収集する（任意、ただし推奨）。

| 理由コード | ラベル |
|-----------|--------|
| `not_applicable` | この利用者には該当しない |
| `already_addressed` | すでに対応済み |
| `insufficient_data` | データ不足で判断できない |
| `disagree` | 判断が異なる |
| `other` | その他（自由記述） |

### 採用確認ダイアログ

採用時の確認ダイアログは以下の構造にする。

```
┌──────────────────────────────────────────┐
│  提案の採用確認                           │
│                                          │
│  以下の変更を適用します：                  │
│                                          │
│  ☑ 支援方針 → 「休憩導線見直し」追記      │
│  ☑ 環境調整 → 「刺激量の調整」追記        │
│  ☐ 目標評価 → 変更なし                   │
│                                          │
│  ──── 適用後の差分プレビュー ────         │
│  [diff 表示エリア]                        │
│                                          │
│  [キャンセル]              [適用して保存]  │
└──────────────────────────────────────────┘
```

### 採用レコード (ProposalAdoptionRecord)

採用・却下の履歴は以下の構造で必ず保存する。

```typescript
interface ProposalAdoptionRecord {
  id: string;
  proposalBundleId: string;
  source: ProposalSource;
  action: 'accepted' | 'dismissed' | 'deferred';
  dismissReason?: string;
  selectedFields: string[];   // 採用時: 実際に反映したフィールド
  decidedBy: string;          // 操作者
  decidedAt: string;          // ISO 8601
  planningSheetId: string;
  provenance: {
    sourceType: ProposalSource;
    sourceIds: string[];
    generatedAt: string;
  };
}
```

## 5. Proposal Lifecycle

> **対応原則:** [原則 6 — 改善接続](principles.md#原則-6--記録は保存で終わらせず改善に接続する)、[原則 8 — 組織知化](principles.md#原則-8--個人最適ではなく組織知化を前提に設計する)

### 提案の状態遷移

```
   pending        ← 生成直後（未確認）
     │
     ├── viewed    ← 画面上で表示された
     │     │
     │     ├── accepted  ← 採用された → 反映 + 履歴保存
     │     │
     │     ├── dismissed ← 却下された → 理由 + 履歴保存
     │     │
     │     └── deferred  ← 保留された → 次回表示時に再提示
     │
     └── expired   ← 有効期限切れ（90 日未操作）→ アーカイブ
```

### 状態に応じた表示

| 状態 | 表示 | Proposal Card の見た目 |
|------|------|----------------------|
| `pending` | アクティブ表示 | 通常カード + アクションボタン |
| `viewed` | アクティブ表示 | 「確認済」バッジ追加 |
| `accepted` | 履歴表示 | ✅ マーク + 反映日時 |
| `dismissed` | 履歴表示 | ❌ マーク + 却下理由 |
| `deferred` | アクティブ表示 | 🕐 マーク + 「保留中」バッジ |
| `expired` | 非表示 | 履歴画面でのみ閲覧可能 |

### 提案の表示優先度

同一利用者に対する複数の提案は、以下の優先度で並べる。

```
1. urgent + pending     （最優先）
2. recommended + pending
3. urgent + deferred
4. suggested + pending
5. recommended + deferred
6. suggested + deferred  （最低優先）
```

## 6. UI Consistency Rules

> **対応原則:** [原則 9 — 一貫性](principles.md#原則-9--機能追加より一貫性を優先する)

### 用語統一

すべての画面で以下の用語を統一する。

| 概念 | 統一用語 | ❌ 使わない表現 |
|------|---------|---------------|
| AI が出す内容 | **候補** / **提案** | 推奨、指示、回答、結論 |
| 内容を取り込む操作 | **採用** | 適用、承認、実行、確定 |
| 内容を退ける操作 | **却下** | 拒否、削除、無視 |
| あとで見る操作 | **保留** | スキップ、パス |
| 根拠・出典 | **根拠** / **出典** | エビデンス、参考、ソース |
| 変更の記録 | **履歴** | ログ、記録 |
| 見直しのきっかけ | **見直し候補** | レビューアイテム、TODO |

### カラーパレット

提案関連 UI で使用するカラーパレットを統一する。

| 用途 | MUI Token | Hex |
|------|-----------|-----|
| 採用済 | `success.main` | `#2e7d32` |
| 却下済 | `error.light` | `#ef5350` |
| 保留中 | `warning.main` | `#ed6c02` |
| 未確認 | `info.main` | `#0288d1` |
| 手動入力 | `grey[500]` | `#9e9e9e` |

### アイコン規約

| 操作 / 状態 | アイコン |
|-------------|---------|
| 提案あり | 💡 (`LightbulbOutlined`) |
| 採用済 | ✅ (`CheckCircle`) |
| 却下済 | ❌ (`Cancel`) |
| 保留中 | 🕐 (`Schedule`) |
| 根拠表示 | 📊 (`BarChart`) |
| 出典リンク | 📎 (`AttachFile`) |
| provenance | 🏷️ (`Label`) |

### レスポンシブ原則

| 画面幅 | Proposal Card の表示 |
|--------|---------------------|
| `≥ 1200px` | 2 カラム表示可 |
| `900px – 1199px` | 1 カラム（フル幅） |
| `< 900px` | スタック表示 + 根拠セクション折りたたみ |

### 新規 UI 追加時のチェックリスト

新しい画面・コンポーネントを追加する際、以下を確認する。

- [ ] 共通の `<ProposalCard>` を使用しているか
- [ ] 用語統一表の用語を使用しているか
- [ ] カラーパレットが統一されているか
- [ ] 3 ボタン標準（採用/却下/後で確認）に従っているか
- [ ] `ProposalAdoptionRecord` が保存されるか
- [ ] Provenance Badge の色分けが統一されているか
- [ ] レスポンシブ対応されているか
- [ ] [設計原則 10 箇条](principles.md) のどの原則に対応するか明記されているか
