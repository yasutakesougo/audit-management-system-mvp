## 0. 目的

本システムの目的は、障害福祉事業所における日常業務を単なる記録システムとしてではなく、**現場の行動を支援し、支援計画と運用改善を循環させるオペレーションOS**として再設計することである。

> [!IMPORTANT]
> 本文書は [設計原則 10 箇条](principles.md) の **原則 10「判断補助 OS」** を画面設計・情報設計レベルで具体化したものである。

### 従来の課題

| # | 課題 | 対応する設計原則 |
|---|------|----------------|
| 1 | 記録入力、支援計画、分析閲覧が分断されている | 原則 6（改善接続） |
| 2 | 利用者や業務に関する情報が複数画面へ散在している | 原則 1（現場導線主体） |
| 3 | 現場職員、支援設計者、管理者に同一UIを適用している | 新規: 役割別非対称UI |
| 4 | システムが「保存先」にはなっても「次の行動」を示していない | 原則 7（日次優先） |
| 5 | 情報過多により認知負荷が高く、入力漏れや見落としが発生する | 原則 2（入力最小化） |

### 本システムの実現目標

* 現場入力の高速化と摩擦削減
* 記録から支援計画への自然な接続
* 管理者向けの例外管理と異常検知
* 役割別に最適化された非対称UI
* 記録・設計・分析が循環する運用基盤

## 1. システムコンセプト

### 1.1 業務フェーズ

本システムは、以下の4つの業務フェーズを支える。

```
1. 現場入力  ←→  Observation Layer（既存アーキテクチャ）
2. 記録蓄積  ←→  Normalization Layer
3. 支援設計  ←→  Insight + Proposal Engine
4. 分析管理  ←→  Knowledge Layer
```

> 既存 7 層パイプライン（[アーキテクチャ §2](support-operations-os-architecture.md#2-concept-model)）との対応関係を明示する。

### 1.2 画面設計レイヤ

業務フェーズを、画面設計上は次の **3 レイヤ** に再構成する。

```
┌──────────────────────────────────────────────────────────────┐
│  A. Execution Layer                                          │
│  今やるべき業務を処理する層                                      │
│  例: Today、日次記録入力、申し送り、アラート確認                    │
│                                                              │
│  ← 既存: Operational UI（Today / Handoff / Daily）              │
├──────────────────────────────────────────────────────────────┤
│  B. Synthesis Layer                                          │
│  蓄積された記録を意味ある支援情報へ変換する層                      │
│  例: 支援計画、モニタリング、提案、ABC分析                        │
│                                                              │
│  ← 既存: Planning Sheet / Monitoring / Proposal Engine          │
├──────────────────────────────────────────────────────────────┤
│  C. Control Layer                                            │
│  施設全体の状態を把握し、例外に介入する層                          │
│  例: 未記入管理、期限超過、インシデント集中、負荷偏在              │
│                                                              │
│  ← 新規: 管理ダッシュボード / 例外管理                            │
└──────────────────────────────────────────────────────────────┘
```

> [!NOTE]
> A層は [原則 1（現場導線主体）](principles.md#原則-1--主役は-ai-ではなく現場導線である) が支配する。
> B層は [原則 3（候補提示）](principles.md#原則-3--提案は命令ではなく候補として提示する) + [原則 4（根拠付与）](principles.md#原則-4--提案には必ず根拠を添える) が支配する。
> C層は新たに定義する **例外駆動管理** 原則が支配する。

## 2. 対象ユーザーと役割別要件

### 2.1 現場職員

| 項目 | 内容 |
|------|------|
| **役割** | 日常支援の実施、支援記録入力、注意事項確認、申し送り確認 |
| **主な利用デバイス** | スマホ、タブレット、一部PC |
| **主な利用レイヤ** | A. Execution Layer |
| **対応原則** | 原則 1（現場導線主体）、原則 2（入力最小化）、原則 7（日次優先） |

#### UI要件

- モバイル最優先（Mobile-First）
- タップ中心 / 片手操作しやすいUI
- 極力タイピング不要（選択式、マクロ、音声入力）
- 入力途中で中断しても復帰可能（自動下書き保存）
- 今日やることが即見える（Today画面直行）

#### 既存実装との対応

| 要件 | 既存実装 | ギャップ |
|------|---------|---------|
| Today直行 | `TodayBentoLayout.tsx` | ✅ 実装済 |
| 申し送り | `HandoffLiveFeed.tsx` | ✅ 実装済 |
| 日中記録 | `TableDailyRecordForm.tsx` | ⚠️ モバイル最適化が必要 |
| 支援ウィザード | `wizard/` | 🔄 進行中 |
| 音声入力 | — | 📋 未着手 |

### 2.2 支援設計者

| 項目 | 内容 |
|------|------|
| **役割** | 個別支援計画作成、モニタリング、記録レビュー、根拠確認、改善提案反映 |
| **主な利用デバイス** | PC |
| **主な利用レイヤ** | B. Synthesis Layer |
| **対応原則** | 原則 3（候補提示）、原則 4（根拠付与）、原則 5（人間決定）、原則 6（改善接続） |

#### UI要件

- 長文入力に強い編集体験
- 過去記録・前回計画・バイタル等を同時参照可能（コンテキストパネル）
- コンテキストスイッチ最小化
- テンプレートやマクロで高速作成
- 候補提案・差分確認・根拠確認ができる

#### 既存実装との対応

| 要件 | 既存実装 | ギャップ |
|------|---------|---------|
| 支援計画編集 | Planning Sheet (`ProvenanceBadge`, `EvidencePatternSummaryCard`) | ✅ 実装済 |
| モニタリング | `GoalProgressCard.tsx`, `IspRecommendationCard.tsx` | ✅ 実装済 |
| コンテキストパネル | — | 📋 **新規必要** |
| テンプレート入力 | — | 📋 未着手 |
| 差分確認 | `DiffViewer` (概念のみ) | 📋 未着手 |

### 2.3 管理者

| 項目 | 内容 |
|------|------|
| **役割** | 施設運営管理、業務遅延把握、異常検知、人員負荷把握、改善アクション実行 |
| **主な利用デバイス** | PC |
| **主な利用レイヤ** | C. Control Layer |
| **対応原則** | 新規: 例外駆動管理 |

#### UI要件

- 全件一覧より **例外表示を優先**

- 赤・橙などの警告色は厳格運用（[UI規約 §6](ui-conventions.md#6-ui-consistency-rules) 準拠）
- 是正行動に直結する導線

| 例外ダッシュボード | — | 📋 **新規必要** |

## 3. 情報設計原則

本システムのUIは、以下の **12 原則** に従う。

> [!NOTE]
> 原則 1〜3 は [設計原則 10 箇条](principles.md) を UI レベルに翻訳したもの。
> 原則 4〜12 は次世代 UI 設計で新たに追加する原則。

| # | 原則 | 説明 | 対応する設計原則 |
|---|------|------|-----------------|
| 1 | 意図駆動の優先順位付け | ユーザーの目的に応じて情報を優先序列化 | 原則 1, 7 |
| 2 | 段階的な情報開示 | Progressive Disclosure で認知負荷を削減 | 原則 2 |
| 3 | コンテキストの近接性 | 関連情報を同一画面内に近接配置 | 原則 6 |
| 4 | 役割に基づく非対称性 | 役割別に最適化されたUI（全員同一にしない） | **新規** |
| 5 | 視覚的トリアージ | 緊急度・重要度による色とサイズの差分 | [UI規約 §2](ui-conventions.md#2-proposal-card-標準) |
| 6 | キーボード＆音声アクセス | キーボードショートカット、音声入力対応 | **新規** |
| 7 | スマートツールによる構造化 | マクロ・テンプレート・タグで入力を構造化 | 原則 2 |
| 8 | 行動指向のエンプティステート | 空状態でも次の行動を促すUI | 原則 7 |
| 9 | システム状態の可視化 | 保存状態・同期状態・エラー状態を常時表示 | **新規** |
| 10 | 検知よりも予防 | 問題発生後の通知より、予兆段階での注意喚起 | 原則 6 |
| 11 | 同期より継続 | オフラインでも入力継続可能 | **新規** |
| 12 | 将来利用を前提とした構造化記録 | AI活用・傾向分析を見据えたデータ構造 | 原則 8 |

### 4.2 Today 画面

> **レイヤ:** A. Execution Layer  
> **対応原則:** 原則 1（現場導線主体）、原則 7（日次優先）  
> **既存実装:** [TodayBentoLayout.tsx](../../src/features/today/layouts/TodayBentoLayout.tsx)

#### 目的

ログイン直後に、**今やるべきことを順序づけて提示する**。

#### 表示ブロック（優先度順）

| # | ブロック | 内容 | 既存実装 |
|---|---------|------|---------|
| 1 | 最優先アラート | 安全・制度上の即対応事項 | `alertRules.ts` → 部分的 |
| 2 | 今日期限の記録 | 未入力の日次記録 | `useNextAction.ts` |
| 3 | 支援計画・モニタリング期限 | 90日サイクルの期限 | `monitoringSchedule` |
| 4 | 申し送り・注意事項 | 引き継ぎ事項 | `HandoffLiveFeed` |
| 5 | 低優先の参照情報 | 参考情報 | — |

#### 設計ルール

- ❌ 静的ダッシュボードにしない
- ✅ タスクは緊急度順に自動ソート
- ✅ 完了したタスクは即座にキューから消える
- ✅ クリックで直接対象画面へ遷移
- ✅ 役割別に表示内容が変わる

#### KPI（Today 画面表示用）

| KPI | 閾値 | 色 |
|-----|------|-----|
| 今日の未入力件数 | >0 で `warning`、>5 で `error` | [UI規約準拠](ui-conventions.md#6-ui-consistency-rules) |
| 今日対応すべきアラート件数 | >0 で `error` | |
| 期限切れタスク件数 | >0 で `error` | |

### 4.5 支援計画画面

> **レイヤ:** B. Synthesis Layer  
> **対応原則:** 原則 3（候補提示）、原則 4（根拠付与）、原則 5（人間決定）  
> **既存実装:** Planning Sheet (`ProvenanceBadge`, `ImportAssessmentDialog`, Bridge 1/2/3)

#### 目的

記録を根拠として支援計画を設計・更新する。

#### レイアウト構造

```
┌──────────┬──────────────────────────────┬──────────────────┐
│ 左:       │  中央: 計画本文編集           │ 右: コンテキスト │
│ セクション │                              │ パネル           │
│ ナビ      │  ┌───────────────────────┐   │                  │
│           │  │ [セクション名]        │   │  過去記録サマリ  │
│ □ 基本情報│  │ [入力フィールド群]    │   │  ヒヤリハット    │
│ ☑ 目標設定│  │ [ProvenanceBadge]     │   │  前回計画        │
│ □ 支援方針│  │ [テンプレ・候補提案]  │   │  アセスメント    │
│ □ 環境調整│  └───────────────────────┘   │  バイタル/傾向   │
│ □ 評価指標│                              │  アンケート特性  │
│           │                              │  モニタリング履歴│
├──────────┴──────────────────────────────┴──────────────────┤
│  下部: 候補提案 / 根拠表示 / 差分表示                       │
│  [ProposalCard] [DiffViewer] [ImportHistory]               │
└────────────────────────────────────────────────────────────┘
```

#### 設計ルール

- 長文編集しやすい（PC最適化）
- セクションごとの完了率表示
- テンプレ・定型文・候補提案（[UI規約 §2](ui-conventions.md#2-proposal-card-標準) 準拠）

## 7. データ要件

### 7.1 記録データの構造化方針

自由記述のみではなく、以下のフィールドを持つ。

```typescript
// 既存の Daily Record スキーマを拡張
interface StructuredDailyRecord {
  // === 既存フィールド ===
  userId: string;
  date: string;           // ISO 8601 date
  recordBody: string;     // 記録本文
  createdBy: string;      // 記録者
  createdAt: string;      // 作成時刻 ISO 8601
  updatedAt: string;      // 更新時刻 ISO 8601

  // === 新規追加フィールド ===
  timeSlot: string;       // 時間帯（朝/午前/昼/午後/夕方）
  activityCategory: string;    // 活動カテゴリ
  specialNotes: string;        // 特記事項
  completionStatus: 'draft' | 'complete';  // 完了状態
  tags: string[];              // 構造化タグ
  riskFlag: boolean;           // リスクフラグ
  relatedPlanKey?: string;     // 支援計画関連キー
  sourceType: 'manual' | 'voice' | 'template';  // ソース種別
}
```

> [!NOTE]
> 既存の `DailyRecord` 型（[src/domain/daily/](../../src/domain/daily/)）との互換性を維持しつつ拡張する。
> SharePoint リスト設計は [SP-Schema AI](/sp-schema) で別途定義する。

---

### 7.2 将来利用前提

記録データは以下に再利用される前提で保存する。

```
記録データ
  ├→ 支援計画生成         ← Bridge 1/2/3（既存）
  ├→ モニタリング         ← Insight Engine（既存）
  ├→ 傾向分析             ← Normalization Layer（既存）
  ├→ インシデント分析     ← Safety Domain（既存）
  ├→ 監査対応             ← Knowledge Layer（既存）
  └→ AI支援候補生成       ← Proposal Engine（Phase 4）
```

## 12. 最終定義

> [!IMPORTANT]
> 本システムの理想形は、帳票を電子化したシステムではない。
> **現場入力、支援設計、管理判断を、役割ごとに最適化されたUIで接続し、次の行動を自然に生み出す福祉オペレーションOS** である。

これは [設計原則 10 箇条](principles.md) の **原則 10** を、画面設計・情報設計レベルで完全に具現化したものである。

```
┌──────────────────────────────────────────────────────────┐
│                  福祉オペレーションOS                       │
│                                                          │
│  ┌────────────────────────────────────────────────────┐  │
│  │  A. Execution Layer                                │  │
│  │  現場職員 → 高速入力 → 次の行動                      │  │
│  ├────────────────────────────────────────────────────┤  │
│  │  B. Synthesis Layer                                │  │
│  │  支援設計者 → 根拠確認 → 計画更新                    │  │
│  ├────────────────────────────────────────────────────┤  │
│  │  C. Control Layer                                  │  │
│  │  管理者 → 例外検知 → 是正行動                        │  │
│  └────────────────────────────────────────────────────┘  │
│                                                          │
│  全レイヤを貫く 7 層パイプライン:                           │
│  Observe → Normalize → Insight → Propose →               │
│  UI → Execute → Knowledge → Loop                         │
│                                                          │
└──────────────────────────────────────────────────────────┘
```
