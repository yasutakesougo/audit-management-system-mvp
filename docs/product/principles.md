## 0. 前提

このシステムは、単なる記録アプリではない。  
目的は、

> **記録を起点に、支援判断・運用改善・知識蓄積までつなぐこと**

である。

したがって、評価基準は「機能数」ではなく、次の **3 つ** で置く。

| # | 評価基準 |
|---|---------|
| 1 | 現場で自然に使われるか |
| 2 | 判断の質を上げるか |
| 3 | 組織の知識として残るか |

---

## 原則 1 — 主役は AI ではなく現場導線である

現場が毎日触るのは、分析画面ではなく **業務導線** である。  
したがって主役は常に、以下のような業務画面でなければならない。

- **Today** — 今日やること
- **Handoff** — 申し送り
- **Daily** — 日中記録
- **Planning** — 支援計画
- **Monitoring** — モニタリング

分析・提案は価値が高くても、前面に出しすぎると現場は構える。

> [!IMPORTANT]
> **表はシンプル、裏は強い。**

---

## 原則 2 — 記録負担を増やしてはならない

分析精度を高めるために入力項目を増やす設計は、長期的に失敗しやすい。  
現場では、入力負担の増加がそのまま **品質低下と離脱** につながるためである。

守るべき方針：

| 方針 | 説明 |
|------|------|
| 入力は最小限 | 必須項目を絞る |
| 選択式を優先 | 自由記述を減らす |
| 既存データを再利用 | 二重入力を防ぐ |
| 自動補完を多用 | 入力速度を上げる |
| 分析側で後から整理 | 後処理で構造化する |

> [!CAUTION]
> **分析のために現場を苦しめない。**

---

## 原則 3 — 提案は命令ではなく候補として提示する

このシステムの提案は、自動決定ではない。  
常に人の判断を支えるための **「候補」** であるべきである。

### ❌ 悪い例

- 「この支援方法に変更してください」
- 「この結論が正解です」

### ✅ 良い例

- 「午後帯の離席増加が見られます」
- 「環境刺激または休憩導線を見直し候補として確認してください」

提案の位置づけ：

- **見直し候補**
- **下書き候補**
- **注意候補**
- **比較材料**

---

## 原則 4 — 提案には必ず根拠を添える

提案が信用される条件は、「当たること」より先に  
**なぜその提案が出たかが分かること** である。

提案には必ず以下を持たせる：

| 必須要素 | 例 |
|---------|-----|
| 対象期間 | 2026-02-01 〜 2026-02-28 |
| 参照記録数 | 日中記録 42 件 |
| 前回比較 | 前月比 +15% |
| 使用ルール | ABC 頻度ルール v2 |
| 出典レコード | record_id: abc-0042 |

> [!IMPORTANT]
> 構造は必ず **「提案 + 根拠 + 参照元」** の 3 点セットにする。  
> これは `provenance`、`proposalBundle`、`audit trail` の中心設計原則となる。

---

## 原則 5 — 最終決定は必ず人が行う

このシステムは **判断補助 OS** であり、自律決定 OS ではない。  
福祉業務においては、特にこの境界が重要である。

### 承認フロー

```text
システムが候補を出す
  → 人が確認する
  → 人が採用・却下する
  → 履歴が残る
```

> **AI 提案 → 人が承認 → 反映 → 履歴保存**

この構造により、以下が成立する：

- ✅ 説明責任
- ✅ 誤提案抑制
- ✅ 監査対応
- ✅ 支援の一貫性

---

## 原則 6 — 記録は保存で終わらせず、改善に接続する

普通のシステムは `記録 → 保存` で終わる。  
しかし Support Operations OS は次の循環を扱う：

```text
記録 → 整理 → 分析 → 提案 → 反映 → 履歴
```

重要なのは、どの画面・どの機能も **「次の改善にどうつながるか」** を持つことである。

| 入力 | 接続先 |
|------|--------|
| 申し送り | → 朝会サマリー |
| 日中記録 | → モニタリング |
| ABC 記録 | → 場面変化検出 |
| モニタリング | → 支援計画見直し |
| 提案採用 | → 知識化 |

---

## 原則 7 — まず日次価値、次に月次価値を作る

大きな理想を先に作るより、毎日の業務で効く価値を先に出す方が定着しやすい。

### 日次（Daily Value）— 優先度：高

| 価値 | 概要 |
|------|------|
| 朝会サマリー | 昨日の要点を自動整理 |
| 申し送り整理 | 引き継ぎ事項の構造化 |
| 今日の注意点 | リスク・変化点の提示 |
| 次アクション導線 | 次にやるべきことへの誘導 |

### 月次（Monthly Value）— 優先度：中

| 価値 | 概要 |
|------|------|
| モニタリング下書き | 記録から自動生成 |
| 目標進捗整理 | 目標達成度の可視化 |
| 支援変更候補 | 見直しが必要な項目の提示 |
| 見直し論点整理 | 議論すべき点の一覧化 |

> [!TIP]
> 現場は毎日使う仕組みを信頼し、その延長で月次支援にも信用を広げる。

---

## 原則 8 — 個人最適ではなく組織知化を前提に設計する

一人のベテランの頭の中にある判断基準を、暗黙知のまま放置してはいけない。  
良い運用は次の形で残すべきである。

- どういう記録が良いか
- どの観点で見直したか
- どの提案を採用したか
- 結果どう改善したか

> [!NOTE]
> 履歴は単なるログではなく、**組織の判断資産** として扱う。

組織知化により強化される領域：

| 領域 | 効果 |
|------|------|
| 新人教育 | 過去の判断事例を学習材料にできる |
| 引き継ぎ | 担当交代時の情報ロスを最小化 |
| 監査説明 | 根拠付きの判断履歴を提示可能 |
| 法人展開 | 他事業所への運用横展開が容易 |

---

## 原則 9 — 機能追加より一貫性を優先する

システムが成長すると、個別に便利な機能を増やしたくなる。  
しかし成功するのは、機能の多さではなく **一貫性** である。

### 一貫性チェックリスト

- [ ] 同じ用語が使われているか
- [ ] 同じ判断フローになっているか
- [ ] 同じ UI 原則で見えるか
- [ ] 同じ履歴構造で追えるか
- [ ] 同じ承認方式で反映されるか

> [!WARNING]
> **バラバラに便利な機能群** ではなく、  
> **同じ思想で動く一つの運用 OS** を目指すべきである。

---

## 原則 10 — このシステムは「記録システム」ではなく「判断補助 OS」である

最終的な定義はここに集約される。

このシステムは、

- ❌ 記録を集めるための箱ではない
- ❌ AI の正解を押し付ける装置でもない
- ❌ 単なるダッシュボードでもない

本質は、

> **人がより良く支援判断を行うための基盤**

### 真の評価軸

| 評価軸 | 問い |
|--------|------|
| 判断速度 | 判断が早くなったか |
| 自然な振り返り | 見直しが自然に起きたか |
| 支援品質の均質化 | 支援の質がそろってきたか |
| 説明可能性 | 根拠を持って説明できるか |
| 知識の蓄積 | 知識が残るようになったか |

## 次段階アクション

この原則を抽象論で終わらせず、**実装ルールに変換する** ため、以下の 3 本を次段階として推進する。

### ① プロダクト憲章化

本ドキュメント（`docs/product/principles.md`）を正式なプロダクト憲章として固定する。  
新機能の設計レビュー時にこの 10 原則への適合を確認する。

### ② UI 設計ルール化

以下の共通 UI 規約を策定し、`docs/product/ui-conventions.md` として文書化する。

- **提案カード** — 候補の表示形式
- **根拠表示** — provenance の見せ方
- **採用 / 却下** — 承認フローの UI パターン

### ③ ロードマップ化

短期・中期・長期の機能優先順位を `docs/product/roadmap.md` として整理し、  
各フェーズが本原則のどれに対応するかを明示する。
