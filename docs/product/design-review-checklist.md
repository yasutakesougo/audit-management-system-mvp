# Support Operations OS — 設計レビュー用チェックリスト

> **ドキュメント種別:** 設計レビュー基準  
> **ステータス:** Draft  
> **参照元:** `docs/product/principles.md`  
> **目的:** 新機能・改修・UI変更が、プロダクト憲章に沿っているかをレビュー時に判定するための実務チェックリスト

---

## 0. 使い方

このチェックリストは、以下の設計レビュー時に使用する。

- 新機能追加
- 既存機能の大幅改修
- UI導線変更
- AI提案 / 自動補完 / 自動整理の追加
- データ構造や履歴構造の変更

レビューでは「便利そうか」ではなく、以下を確認する。

1. 現場で自然に使われるか
2. 判断の質を上げるか
3. 組織知として残るか

---

## 1. レビュー判定ルール

各項目を以下で判定する。

- **OK**: 原則に適合している
- **WARN**: 方向性は合うが、補強が必要
- **NG**: 原則に反しており、そのまま進めるべきではない
- **N/A**: 今回の変更では対象外

### レビュー通過条件

- **NG が 0 件**
- **WARN は理由と対応方針を記録**
- AI提案・自動化・承認フロー変更を含む場合は、必ず「原則3・4・5」を通す
- 履歴構造や保存方式を変える場合は、必ず「原則6・8・9」を通す

---

## 2. 3つの最上位判定

### A. 現場で自然に使われるか

- [ ] 現場の主導線（Today / Handoff / Daily / Planning / Monitoring）を壊していない
- [ ] 画面を開いた瞬間に、次にやることが分かる
- [ ] 現場が「考え込まないと使えないUI」になっていない
- [ ] 分析やAIが前面に出すぎて、現場を構えさせていない
- [ ] 1日の業務の流れの中で自然に使える

判定: OK / WARN / NG / N/A  
コメント:

---

### B. 判断の質を上げるか

- [ ] 単なる情報追加ではなく、判断を助ける形になっている
- [ ] 注意点・比較・候補・優先順位など、次の判断につながる
- [ ] 根拠なしの提案やブラックボックスな表示になっていない
- [ ] 人が確認しやすい粒度で表示されている
- [ ] 誤解を招く断定表現を避けている

判定: OK / WARN / NG / N/A

---

### C. 組織の知識として残るか

- [ ] 誰が何を見てどう判断したかが後から追える
- [ ] 採用 / 却下 / 修正の履歴が残せる
- [ ] 個人の頭の中だけで完結しない
- [ ] 新人教育や引き継ぎに使える形で残る
- [ ] 将来の監査説明や横展開に耐えられる

判定: OK / WARN / NG / N/A

## 3. 原則別チェック

### 原則1 — 主役は AI ではなく現場導線である

- [ ] 主役が「分析画面」ではなく「業務導線」になっている
- [ ] Today / Handoff / Daily / Planning / Monitoring のどれに効くか明確
- [ ] 現場作業の邪魔をしない
- [ ] 裏側が高度でも、表側はシンプルである

**NG例**
- 分析のためだけの新画面を増やす
- 現場入力前に長い解釈画面を挟む

判定: OK / WARN / NG / N/A  
コメント:

---

### 原則2 — 記録負担を増やしてはならない

- [ ] 必須入力が増えすぎていない
- [ ] 自由記述を安易に追加していない
- [ ] 既存データ再利用で二重入力を避けている
- [ ] 自動補完・初期値・選択式を使っている
- [ ] 分析都合を入力側へ押し付けていない

**NG例**
- 分析精度向上のためだけに新たな必須項目を増やす
- 同じ情報を複数画面で再入力させる

判定: OK / WARN / NG / N/A  
コメント:

---

### 原則3 — 提案は命令ではなく候補として提示する

- [ ] 表現が断定ではなく候補になっている
- [ ] 「見直し候補」「下書き候補」などの位置づけが明確
- [ ] ユーザーが却下できる
- [ ] システムが勝手に決定したように見えない
- [ ] UI上で人の判断余地が確保されている

**NG例**
- 「この支援方法に変更してください」
- 「この結論が正解です」

判定: OK / WARN / NG / N/A  
コメント:

---

### 原則4 — 提案には必ず根拠を添える

- [ ] 提案に対象期間がある
- [ ] 提案に参照件数や比較情報がある
- [ ] 提案にルール名または生成根拠がある
- [ ] 提案に参照元レコードまたは provenance がある
- [ ] 「提案 + 根拠 + 参照元」の3点セットを満たす

**NG例**
- 理由不明のおすすめ表示
- 出典不明のスコア表示

判定: OK / WARN / NG / N/A  
コメント:

---

### 原則5 — 最終決定は必ず人が行う

- [ ] 採用 / 却下 / 保留を人が選べる
- [ ] 自動反映される場合、その範囲が限定されている
- [ ] 承認フローがUI上で分かる
- [ ] 誰が承認したかが残る
- [ ] 説明責任を人が持てる構造である

**NG例**
- AI提案が自動で記録や計画に反映される
- 承認者不明のまま状態が更新される

判定: OK / WARN / NG / N/A

### 原則6 — 記録は保存で終わらせず、改善に接続する

- [ ] 入力の先に「次の活用先」が定義されている
- [ ] 記録がサマリー・モニタリング・計画見直しに接続する
- [ ] 保存しただけで埋もれない
- [ ] 次アクションや次画面への導線がある
- [ ] 反映結果が履歴として戻ってくる

判定: OK / WARN / NG / N/A

### 原則7 — まず日次価値、次に月次価値を作る

- [ ] 今回の変更が日々の現場価値に効く
- [ ] いきなり重い月次機能を優先していない
- [ ] 毎日使う機能の改善になっている
- [ ] 現場が価値を体感しやすい
- [ ] 月次価値は日次価値の延長として設計されている

判定: OK / WARN / NG / N/A

### 原則8 — 個人最適ではなく組織知化を前提に設計する

- [ ] 良い判断や良い記録が再利用できる
- [ ] 個人の経験が履歴として残る
- [ ] 他職員が見ても分かる形になっている
- [ ] 引き継ぎや教育に転用できる
- [ ] 組織全体で使う前提の命名・構造になっている

判定: OK / WARN / NG / N/A

### 原則9 — 機能追加より一貫性を優先する

- [ ] 既存用語と矛盾しない
- [ ] 既存承認フローと矛盾しない
- [ ] UIパターンが既存と揃っている
- [ ] 履歴構造が既存と揃っている
- [ ] 今回だけの例外設計を作っていない

**NG例**
- 同じ意味なのに画面ごとに用語が違う
- 似た提案機能なのに画面ごとに承認方法が違う

## 4. 追加ゲート（重要変更向け）

### AI / 提案 / 自動生成を含む場合の追加確認

- [ ] 候補提示に留まっている
- [ ] 根拠が表示される
- [ ] 参照元に辿れる
- [ ] 人の承認なしに本番反映しない
- [ ] 誤提案時の影響が限定されている

---

### データ構造 / 履歴 / 保存方式変更時の追加確認

- [ ] 後から追跡可能である
- [ ] 既存履歴との整合性が保たれる
- [ ] 監査説明に耐えられる
- [ ] 組織知として再利用できる
- [ ] 既存画面や既存運用との一貫性がある

---

### 運用品質 / レジリエンス確認

- [ ] 障害時に気づける
- [ ] 誤作動時に原因を追いやすい
- [ ] 部分失敗でも全体停止しにくい
- [ ] 自動化失敗時に人が介入できる
- [ ] 「動く」だけでなく「運用で扱える」設計になっている
