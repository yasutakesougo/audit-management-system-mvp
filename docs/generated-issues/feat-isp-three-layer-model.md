# feat: ISP と支援計画シート等を分離した三層モデルをシステム設計の前提にする

## 背景

現場運用と制度要件の両面から、個別支援計画（ISP）と支援計画シート等は同一文書として扱えない。
ISPは法定の中核文書であり、本人意向・支援方針・課題・目標・達成時期・同意・交付・モニタリング・見直しを管理する上位計画である。
一方、支援計画シート等は、行動特性に応じた場面別・工程別の具体支援と、その実施記録を担う実装文書である。

現状の設計・今後の機能追加において、この区別を曖昧にすると制度不適合・監査弱さ・現場運用の破綻が起きるため、システム根幹の設計原則として明文化する。

### 既存コードベースの現状

既存コードでは部分的に三層構造が反映されている：

| 層 | 既存実装 | 状態 |
|---|---|---|
| 第1層 ISP | `SupportPlans` SPリスト, `ispGoalMapper.ts`, ISP Editor | ✅ 基本構造あり |
| 第2層 支援計画シート | `ibdTypes.ts` → `SupportPlanSheet` | ✅ 型定義あり |
| 第3層 支援手順書兼記録 | `ibdTypes.ts` → `SupportProcedureManual` | ⚠️ 型あり、実施記録機能は未完 |
| 相互参照 | `ISPReference` | ⚠️ スナップショット参照のみ |
| 状態管理 | ISP: `draft/confirmed/obsolete` | ⚠️ 制度要件の状態遷移は未実装 |
| 監査証跡 | `SPSHistoryEntry` | ⚠️ 同意・交付・モニタリング証跡は未実装 |

## 目的

- ISP と支援計画シート等を別レイヤーとして明確化する
- 両者を相互参照可能な構造にする
- 支援手順書兼記録を単なる日報ではなく実施ログとして扱う
- 監査耐性、制度適合性、再現性を高める

## 要件

### 1. 三層モデル

#### 第1層: ISP

- 本人・家族の意向
- 総合的支援方針
- QOL向上課題
- 長期目標・短期目標
- 達成時期
- 留意事項
- 同意
- 交付
- モニタリング
- 見直し

#### 第2層: 支援計画シート

- 行動観察
- 情報収集
- 分析・仮説
- 支援課題
- 対応方針
- 環境調整
- 関わり方の具体策
- 版管理
- 見直し日

#### 第3層: 支援手順書兼記録

- 時間帯
- 活動
- 支援手順
- 実施チェック
- 様子
- 特記事項
- 連絡事項
- 実施者
- 実施日時

### 2. リンク構造

- 1 ISP : N 支援計画シート
- 1 支援計画シート : N 支援手順記録
- 相互参照可能
- 上位目標から具体支援へ辿れること
- 記録から根拠となる支援計画・ISPへ戻れること

### 3. 監査証跡

- 作成者
- 更新者
- 作成日
- 更新日
- 版番号
- 会議記録
- 同意記録
- 交付記録
- モニタリング記録
- 見直し履歴

### 4. 状態管理

#### ISP

- アセスメント
- 原案作成
- 会議
- 説明
- 同意取得
- 交付
- 実施
- モニタリング
- 見直し

#### 支援計画シート等

- アセスメント
- 仮説整理
- 計画作成
- 手順化
- 実施記録
- 再アセスメント
- 改訂

## 禁止事項

- ISP と支援計画シートの一本化
- 支援計画シートのメモ化
- 支援手順記録の日報化
- 履歴なし更新
- 同意・交付・会議・モニタリング抜けを許す設計
