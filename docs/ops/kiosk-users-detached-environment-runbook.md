# `/kiosk/users` 複製環境 runbook

## 1. 目的

このrunbookは、`/kiosk/users` 支援手順記録を別事業所で試行する前に、複製環境の分離条件と運用手順を確認するための手順書である。

この手順は、#2360 の複製環境試行方針に従う。

今回の範囲はdocs-onlyである。実装、テスト追加、デプロイ、SharePoint作成、SharePoint列追加、本番データ変更、環境変数変更、シークレット変更、PDF生成は行わない。

試行記録は、原則として正式記録ではなく、紙記録または既存運用の補助記録として扱う。

## 2. 完全分離の前提

試行環境は、既存本番とは別URL、別SharePoint、別認証、別データで分離する。

次のいずれかが既存本番と共有されている場合は、試行を開始しない。

- デプロイURL
- SharePointサイトまたはSharePointリスト群
- 利用者マスタ
- 支援手順マスタ
- 実施記録
- 認証対象職員
- API権限
- 環境変数
- シークレット
- 管理者
- 問い合わせ窓口
- 運用ログの確認先

GitHub fork、別branch、別repoだけでは分離条件を満たさない。データ、認証、デプロイ、SharePointが分かれていることを必須条件とする。

## 4. 別URL確認

別URLは、既存本番と利用者、職員、管理者が取り違えない状態にする。

確認する項目は次とする。

- 既存本番URLと試行URLが異なる。
- ブックマーク、案内文、QRコード、端末ショートカットが試行用に分かれている。
- 試行URLから既存本番の利用者情報や支援記録が表示されない。
- 既存本番URLから試行データが表示されない。
- 問い合わせ時に、職員が本番と試行を区別して説明できる。

URLが同じ、または職員が本番と試行を区別できない場合は、試行を開始しない。

## 5. 別SharePoint確認

試行環境では、既存本番のSharePointサイト、リスト、列、権限、Power Automate、監査ログ運用へ書き込まない。

確認する項目は次とする。

- 試行用のSharePointサイトまたはリスト群が既存本番と分かれている。
- 利用者マスタ、支援手順マスタ、実施記録の保存先が試行用である。
- 既存本番リストに試行データを置いていない。
- 試行用リストの権限が対象職員に限定されている。
- 既存本番のPower Automateや集計処理に試行データが流れない。

SharePoint作成、列追加、権限変更はこのPRでは行わない。実作業に進む場合は、別PRまたは別作業として責任者と変更内容を固定する。

## 6. 別認証確認

試行環境の認証対象は、対象事業所の職員に限定する。

確認する項目は次とする。

- 対象外職員が試行環境へ入れない。
- 既存事業所の利用者情報や支援記録を閲覧できない。
- 試行事業所の職員が既存本番データへアクセスできない。
- 管理者権限を持つ人が明確である。
- 退職、異動、試行終了時のアクセス停止手順が決まっている。

メニュー非表示や画面上の案内だけを認証分離の代替にしない。認証、権限、接続先を分けて確認する。

## 7. 別データ確認

試行データは、既存本番データと混ぜない。

確認する項目は次とする。

- 利用者IDが既存本番と衝突しない。
- 事業所コード、`facilityId` 相当、`sourceSystem` 相当の識別情報を残す。
- 支援手順ID、記録ID、作成者、確認者、承認者が試行環境内で追跡できる。
- 紙記録との対応関係を残す。
- 将来統合する場合でも、そのまま本番へ投入しない前提を共有している。

将来、法人共通化へ戻す場合は、#2358 の多事業所化方針とは別に、ID、権限、SharePoint filter、URL、repository、テスト境界を棚卸ししてから移行可否を判断する。

## 8. 作業順序

試行準備は次の順序で進める。

1. 対象事業所、対象利用者、対象職員を決める。
2. 紙記録を棚卸しし、データ化対象と対象外を分ける。
3. 入力者、確認者、承認者を決める。
4. 利用者マスタと支援手順マスタを紙記録からデータ化する。
5. 別URL、別SharePoint、別認証、別データの分離条件を確認する。
6. 障害時に紙記録へ戻る手順を確認する。
7. 試行開始前チェックをすべて `OK` にする。
8. 最小人数で試行を開始する。
9. 試行中の記録、問い合わせ、停止条件を毎週確認する。

入力完了だけでは試行開始しない。確認者または承認者の確認後に、試行対象として扱う。

## 9. 紙記録併用ルール

試行中も紙記録を主記録として継続する。

紙記録併用のルールは次とする。

- 紙記録の記入を止めない。
- 試行環境の記録は補助記録として扱う。
- 紙記録と試行記録の不一致を見つけた場合は、紙記録を優先して確認する。
- 不一致の原因、確認者、対応日を残す。
- 紙記録の廃止、正式記録化、保存年限の変更はこのrunbookでは扱わない。

紙記録へ戻れない運用になった場合は、試行を一時停止する。

## 10. 試行中の記録ルール

試行中の記録は、事実、観察、実施した支援、本人の反応を分ける。

記録に残す内容は次に限定する。

- 実施日時
- 対象利用者
- 支援手順
- 実施した支援
- 本人の観察可能な反応
- 紙記録との対応関係
- 入力者
- 確認が必要な事項

人格評価、推測、責任追及に見える表現は避ける。必要な場合は、具体的な行動や観察事実へ置き換える。

## 11. 問題発生時の停止条件

次のいずれかが起きた場合は、試行を停止または一時停止する。

- 既存本番データ、別事業所データ、対象外利用者データが表示される。
- 対象外職員が試行環境へ入れる。
- 実施記録の責任者、確認者、修正責任が不明になる。
- 紙記録との二重運用で現場負担が過大になる。
- 支援手順の更新、確認、承認が止まる。
- 障害時に紙記録へ戻れない。
- SharePoint、認証、URL、データの分離条件を説明できない。
- 本番側へ試行データを書き込んだ疑いがある。

停止した場合は、対象範囲、発生日、確認者、一次対応、再開条件を記録する。

## 12. 切り戻し方針

切り戻し時は、紙記録運用へ戻る。

切り戻し手順は次とする。

1. 試行環境の新規入力を止める。
2. 対象職員へ紙記録へ戻すことを伝える。
3. 試行中に作成された記録を紙記録と突合する。
4. 不一致、未確認、承認待ちの記録を一覧化する。
5. 試行環境のデータを本番へ移さない。
6. 再開する場合は、停止原因と再開条件を確認する。

切り戻しは、試行環境の削除やデータ削除を意味しない。削除、保存、監査対応は別途判断する。

## 13. 本番側へ戻さない事項

次は、このrunbookの範囲では本番側へ戻さない。

- 試行環境で作成した利用者ID
- 試行環境で作成した支援手順ID
- 試行環境で作成した実施記録
- 試行用SharePoint設定
- 試行用認証設定
- 試行用環境変数
- 試行用シークレット
- 試行中に作成した一時運用ルール

本番へ戻す必要が出た場合は、法人共通化、監査、保存年限、責任者、訂正履歴、`facilityId` 境界を別途確認する。
