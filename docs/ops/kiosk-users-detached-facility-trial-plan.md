# `/kiosk/users` 支援手順記録 複製環境による別事業所試行計画

## 1. 結論

別事業所での初期試行は、既存本番へ追加しない。

別URL、別SharePoint、別認証、別データで完全分離した複製環境として扱う。

これは、法人共通システム化ではない。

紙記録運用の事業所で、支援手順記録を限定的に試すための隔離試行である。

## 2. #2358 との関係

#2358 は、同一システム内で多事業所化する場合の方針整理である。

今回の計画は、それとは別トラックである。

#2358 では `facilityId` 境界が必要と整理している。

今回の複製環境では、短期試行に限り、物理分離によって混入リスクを下げる。

ただし、将来統合する可能性があるため、データには `OrgCode`、`OrgName`、`facilityId`、`sourceSystem` 相当の識別情報を持たせる方針とする。

将来、法人共通化へ戻す場合は、複製環境で作成したデータをそのまま本番へ混ぜない。利用者ID、事業所ID、支援手順ID、記録ID、作成者、承認者、更新日、紙記録との対応関係を棚卸しし、`facilityId` 境界が型、repository、URL、テスト、SharePoint filter に入っていることを確認してから移行可否を判断する。

## 3. 試行の前提

対象事業所は、現在システム管理されていない。

利用者情報、支援手順、実施記録は紙記録から新規にデータ化する。

既存事業所のSharePoint、利用者データ、支援記録とは共有しない。

試行中の記録は、原則として正式記録ではなく、紙記録または既存運用の補助記録として扱う。

正式記録として扱う場合は、監査、保存年限、責任者、訂正履歴、紙記録廃止条件を別途決める。それまでは紙記録を主記録として継続する。

## 4. 完全分離する対象

次を既存本番と分離する。

- デプロイURL
- SharePointサイトまたはSharePointリスト群
- 利用者マスタ
- 支援手順マスタ
- 実施記録
- 認証対象職員
- 環境変数
- API権限
- シークレット
- 管理者
- 運用runbook
- 問い合わせ窓口

GitHub fork または別repoは候補とする。

ただし、GitHub forkだけでは分離にならない。

データ、認証、デプロイ、SharePointが分かれていることを必須条件とする。

環境分離では、少なくともデプロイ先、設定キー、接続先URL、認証設定、API permission、シークレット管理者、運用ログの確認先を既存本番と分ける。

SharePoint分離では、既存本番のサイト、リスト、列、権限、Power Automate、監査ログ運用へ書き込まない。別サイトまたは別リスト群のどちらを使う場合でも、既存本番リストを試行データ置き場にしない。

認証・権限分離では、対象事業所の職員だけを試行環境へ招待し、既存事業所の利用者情報や支援記録を閲覧できない状態を試行開始条件にする。

## 5. 試行対象機能

初期試行の対象は `/kiosk/users` 支援手順記録に限定する。

対象に含める機能は次とする。

- 利用者選択
- 支援手順一覧
- 支援手順詳細
- 実施記録
- 履歴確認
- 実施済み判定

初期試行の対象外は次とする。

- 請求
- PDF
- 法人横断集計
- 個別支援計画本体
- 監査向け正式記録化
- 他事業所連携
- 既存本番データとの統合
- 既存本番SharePointへの書き込み

本格展開へ進む判断は、初期試行の入力負担、記録品質、権限事故の有無、手順更新責任、紙記録との整合、障害時の代替手段が確認できてから行う。

## 6. 不要機能の扱い

複製環境では、不要機能をできるだけ表示しない。

ただし、初期docs段階では実装しない。

実装に進む場合は、feature flag、環境変数、ルート制御、メニュー制御のどれで隠すかを別PRで判断する。

不要機能を非表示にしても、認可やデータ分離の代替にはしない。非表示は誤操作防止であり、権限分離、接続先分離、SharePoint分離を必須条件とする。

## 7. 紙記録からデータ化する対象

最低限、次をデータ化する。

- 利用者表示名
- 利用者ID
- 所属事業所
- 有効/無効
- 支援手順名
- 支援手順の本文
- 実施タイミング
- 注意事項
- 禁忌事項
- 手順作成者
- 手順確認者
- 手順承認者
- 更新日
- 紙記録との対応関係
- 記録に残してよい表現

紙記録から転記する際は、原文をそのままデジタル化してよい情報と、記録表現として修正が必要な情報を分ける。

## 8. データ化の役割分担

紙記録からのデータ化では、入力担当者と確認担当者を分ける。

- 入力者: 紙記録をデータ化する人
- 確認者: 支援内容を理解している現場責任者
- 承認者: 管理者またはサービス管理責任者

支援手順は古くなると危険である。

そのため、入力完了だけでは試行開始しない。

確認者または承認者の確認後に試行対象とする。

入力、確認、承認の担当者が同一人物になる場合は、試行前にリスクを明示し、少なくとも現場責任者による最終確認を残す。

## 9. 最小試行スコープ

最小試行は次の範囲とする。

- 対象事業所: 1事業所
- 対象利用者: 3〜5名
- 対象職員: 限定
- 対象機能: 支援手順の閲覧と実施記録
- 試行期間: 1〜2か月
- 記録位置づけ: 紙記録の補助
- 代替手段: 紙記録を継続
- 中止条件: 他事業所データ混入、記録責任不明、入力負担過多、手順更新責任不明

次のいずれかが起きた場合は、試行を中止または一時停止する。

- 既存本番データ、別事業所データ、対象外利用者データが表示される。
- 実施記録の責任者、確認者、修正責任が不明になる。
- 紙記録との二重運用で現場負担が過大になる。
- 支援手順の更新・承認が止まり、古い手順を参照する状態になる。
- 障害時に紙記録へ戻れない。
- 認証対象外の職員が試行環境へ入れる。

## 10. 工数目安

最小試行の目安は 10〜18人日とする。

1事業所・少人数で安全試行する場合は 18〜30人日とする。

主要利用者を一通り登録する場合は 25〜45人日とする。

正式記録化や法人統合を含める場合は 45人日以上の別プロジェクトとする。

工数には、紙記録の読み取り、入力、確認、承認、SharePoint分離、認証分離、不要機能の非表示、試行runbook、職員説明、初回運用立ち会いを含める。

## 11. 実装に進む前の停止条件

次が未確定なら、実装に進まない。

- 対象事業所
- 対象利用者
- 対象職員
- 管理者
- 問い合わせ窓口
- 紙記録との関係
- 正式記録か補助記録か
- SharePoint分離方式
- 認証対象者
- シークレット管理者
- デプロイ責任者
- データ化の入力者
- データ化の確認者
- 支援手順の承認者
- 障害時の代替手段
- 試行終了後の扱い

本格展開へ進む場合は、さらに次を確認する。

- 試行期間中に他事業所データ混入がない。
- 紙記録との突合で重大な不一致がない。
- 職員が実施記録を日常業務内で入力できる。
- 支援手順の更新責任者と承認者が継続運用できる。
- 認証、SharePoint、デプロイ、運用runbookの分離が説明可能である。
- 法人共通化する場合の `facilityId` 境界実装方針が決まっている。

## 12. 実装PR分割案

今回は実装しない。

実装する場合は、次のように小PRに分ける。

1. docs: 複製環境による別事業所試行計画を固定
2. docs: 紙記録データ化テンプレートを追加
3. docs: 複製環境runbookを追加
4. chore: 別環境の設定キー一覧を整理
5. feat: 不要機能をfeature flagで非表示
6. chore: 別デプロイ環境を追加
7. chore: 別SharePoint接続設定を追加
8. test: 複製環境のsmoke確認を追加
9. docs: 試行開始チェックリストを追加

各PRでは、既存本番、#2358、他の未完了PRへ混入させない。

## 14. 次の推奨アクション

まず、このdocs-only PRで方針を固定する。

その後、法人側に対象事業所、対象利用者、対象職員、正式記録か補助記録か、SharePoint分離方式を確認する。

確認が済んだら、紙記録データ化テンプレートと複製環境runbookを別PRで作成する。

次のPRに進む前に、今回の複製環境試行が法人共通システム化ではないことを関係者へ明示する。
