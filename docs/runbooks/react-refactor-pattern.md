# React Conservancy: 巨大ファイル・責務混在の分離パターン (Nightly Patrol標準手法)

本Runbookは、Nightly Patrol等で発見された「肥大化ファイル（600行超）」「テストカバレッジの不足」「保守・変更が困難なコンポーネント」を、**ビジネス要件や既存の挙動を変えずに安全に解体・テスト保護する** ための標準化された再現可能なプロセスを定めます。

## 🎯 判断基準と適用ルート

対象ファイルのコード構造を確認し、以下の2パターンのどちらに該当するかを瞬時に判別します。

### Pattern A: 【Logic混在型】（UIコンポーネントに副作用や複雑な計算ロジックが同居している）
**👉 実行ルート: `/test-design` で純関数化・仕様固定 → その後 `/refactor`**

* **STEP 1 (`/test-design`)**:
  - UIや非同期フックから完全に切り離せる、純粋な判定・集計・フィルタ・変換ロジックを特定する。
  - そのロジックを `logic/` などへ抽出したと想定し、既存の挙動（バグやエッジケースへのフォールバック等も含む）をそのまま正として網羅的な単体テストを書き、振る舞いを「固定」する。
* **STEP 2 (`/refactor`)**:
  - テストによって守られた安全網の上で、物理的なファイル分割（ロジックモジュール化とインポートへの差し替え）を実施する。

*(例: `operation-hub` のカスタムフック、`ExceptionTable.tsx` の表示行生成ロジック)*

---

### Pattern B: 【UI同居型】（すでにDomain層やHookでロジックが分離されており、単にUIが長大になっている）
**👉 実行ルート: `/test-design` をスキップ → 即 `/refactor` 直行**

* **STEP 1 (`/refactor` 直行)**:
  - 状態管理や複雑な計算処理が既になされている（Dumb化されている）ことを確認する。
  - 親コンポーネントと同居しているサブセクション、カード類、ヘッダー等のUI部品を、`/components` 配下のサブディレクトリ（例: `/components/hub/`）へ機械的に解体・別ファイル化する。
  - **原則**: 各コンポーネントには必要なPropsのみを最小限で渡す。色の定義などUI系定数は `constants.ts` へ完全純化する。

*(例: `MonitoringDailyDashboard.tsx`, `UserDetailPage.tsx`)*

---

## 🔒 守るべき厳格なルール (Guardrails)

1. **既存挙動を変更しないこと**
   - 目的は「債務返済（保守可能な状態にすること）」であり「改善・機能追加」ではない。不格好な判定条件もそのまま維持する。
2. **Domain ロジックを後から触らないこと**
   - `Pattern A` においては、UI分割時にドメイン計算を絶対にイジらない。修正するならテストが通るかどうかで絶対の担保を取る。
3. **新規バレルファイル（`index.ts`による再エクスポート集約）を作成しないこと**
   - 不必要な循環依存やバンドル肥大化を防ぐため、物理分割したファイルからの Named Export を直接インポートさせる。
4. **競合対応時は `origin/main`（最新の実装要件）を優先すること**
   - 抽出と同時に他者が新機能を入れたことによる Conflicts は、新機能側を残し、そこに分離したUI・Logicを当てはめるように解決する。
