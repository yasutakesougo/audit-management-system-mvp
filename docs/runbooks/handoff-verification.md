# Handoff モジュール検証手順 (Runbook)

Handoff機能が正しく実装されていること（一覧・追加・既読化）を確認するための手順書です。

## 1. 動作モードと環境変数

Handoff機能は、環境変数 `VITE_SP_ENABLED` に基づいて自動でフェッチ先を切り替えます。  
運用上のブレを防ぐため、以下のルールで統一します。

- **SharePoint モード**: `VITE_SP_ENABLED=true`
- **Local モード**: それ以外（未定義、または `false` など）

## 2. Local モードの確認シナリオ

Local モードでは、ブラウザメモリ上のモックデータに対して CRUD が正常に行えるかを確認します。

### 2.1 パネルの初期表示
1. `VITE_SP_ENABLED` が未設定または `false` の状態で `npm run dev` を起動する。
2. `/today` (今日の業務画面) にアクセスする。
3. 画面下段に「今日の申し送り」パネルが表示されていることを確認する。
4. 初期状態では「本日の申し送りはありません。」と表示されるか、初期モックがある場合は一覧表示されることを確認する。

### 2.2 新規追加フロー
1. Handoff パネル右上の「申し送り追加」ボタンをクリックする。
2. モーダルが開くことを確認し、以下の内容を入力する：
   - 対象者名: `山田 太郎`
   - 優先度: `通常`
   - 記録者名: `テスト職員1`
   - 本文: `テストの申し送り内容です`
3. 「保存」ボタンをクリックする。
4. 成功トーストが表示され、パネルの一覧に追加された内容が即座に反映されていることを確認する。

### 2.3 Priority と Unread の表示確認
1. 追加されたアイテムが「未読」ラベルを持ち、強調表示されていることを確認する。
2. レベルを変えて「重要」「緊急」の優先度で新規追加し、それぞれ対応するラベル（重要・緊急）や異なる背景色で強調表示されることを確認する。
3. 対象者名と記録者名がカード内に正しく表示されていることを確認する。

### 2.4 Status 変化の確認 (既読化)
1. 未読のアイテムの右下にある「確認済み」ボタンをクリックする。
2. アイテムが「既読」状態になり、強調表示（未読ラベルや背景色）が解除されることを確認する。
3. 「確認済み」ボタンが非表示になっていることを確認する。

## 3. SharePoint モードの確認シナリオ

SharePoint モードでは、SharePoint REST API を介して実際のリストと通信します。

### 3.1 事前準備
1. SharePoint 上に `SupportRecord_Handoff` などの該当リストを作成し、内部定義通りのカラムがあるか確認する。

### 3.2 動作と永続性の確認
1. `VITE_SP_ENABLED=true` を設定して `npm run dev` を再起動する。
2. Local モードと同様の手順で、一覧表示、追加、既読化が画面上のエラーなく完了することを確認する。
3. 変更を加えた後ブラウザをリロードし、SharePoint から再度データがフェッチされて状態が一貫して復元（永続化）されていることを確認する。

## 4. UI 調整のチェックポイント
ローカル環境で検証中、以下の点を併せて確認し、必要により微細なコンポーネント調整を行ってください。
- データ空状態のメッセージ表示
- 優先度の文言・アイコン表現
- unread(未読) / read(既読) 時の色の視認性
- 対象者名を空白で保存した際のフォールバック表示
