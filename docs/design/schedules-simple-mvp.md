# Simple Schedule MVP 設計書

作成日: 2026-07-13
位置づけ: スケジュール機能の初期実装スコープ基準
関連資料: `docs/design/schedules-to-be-design.md`

## 1. 目的

Simple Schedule MVP の目的は、次の一文に集約する。

> 職員が、今日と今週の予定、担当、変更点を迷わず確認できるようにする。

この設計書は、将来構想としてのTo-Be設計を否定しない。To-Be設計は将来拡張の参照資料として維持し、本書は最初に実装する範囲を意図的に小さく保つためのスコープ制御文書とする。

## 2. MVPに含める判断基準

MVPに含める機能は、次の3条件をすべて満たすものだけとする。

| 条件 | 内容 |
| --- | --- |
| 日常利用 | 職員が日常的に使う。 |
| 行動判断 | 今日または今週の行動判断に直接必要。 |
| 低い入力負担 | 追加しても入力手順や例外処理が大きく増えない。 |

次のいずれかに該当する機能は、MVPには入れずTo-Be側に残す。

| 除外条件 | 例 |
| --- | --- |
| 管理者だけが使う | 通知ポリシー管理、監査出力 |
| 月数回以下しか使わない | 高度な会議運用、特殊な締切管理 |
| 例外処理が多い | 定期予定、祝日調整、系列変更 |
| 他システム同期が必要 | 送迎システム自動同期、外部カレンダー双方向同期 |
| 個別設定が必要 | 利用者別表示設定、個人別通知設定 |

## 3. MVP画面

MVPの画面は実質2画面に絞る。

### 3.1 今日の予定

職員が最初に見る画面である。目的は、今日やること、担当、注意点、変更点をすぐ確認できるようにすること。

表示項目は以下に限定する。

| 項目 | 説明 |
| --- | --- |
| 時刻 | 開始時刻。終了時刻はある場合のみ表示する。 |
| 予定名 | 短いタイトル。 |
| 対象者または担当 | 利用者名、担当職員名、またはチーム名。 |
| 場所 | 未設定なら表示しない。 |
| 注意事項 | メモから短く表示する。 |
| 変更あり | 変更表示の契約に従って表示する。 |

編集機能、詳細な状態、連携ID、同期状態は通常表示に出さない。

### 3.2 週表示

週表示を標準の予定一覧とする。目的は、今週の予定を確認し、必要な単発予定を登録・編集・中止できるようにすること。

初期フィルタは次の3つだけにする。

| フィルタ | 意味 |
| --- | --- |
| 全体 | 組織またはチーム全体の予定。 |
| 自分 | 自分が担当または関係する予定。 |
| 利用者 | 利用者に紐づく予定。 |

日表示と月表示はMVPには含めない。後続候補として扱う。

## 4. 予定登録・編集

### 4.1 入力項目

MVPでは、登録フォームの初期表示を最小限にする。

| 項目 | 必須 | 備考 |
| --- | --- | --- |
| title | Yes | 予定名。 |
| date | Yes | 予定日。 |
| start | Yes | 開始時刻。 |
| end | No | 終了時刻。業務要件が確定した場合のみ必須化する。 |
| target | No | 利用者、職員、チームなど。 |
| assignee | No | 担当者。未定を許容する。 |
| location | No | 場所。 |
| memo | No | 注意事項や補足。 |

送迎、会議、繰り返し、通知、外部連携に関する項目は初期表示しない。

### 4.2 操作メニュー

既存予定に対する操作は、予定カードまたは編集画面の操作メニューに置く。

| 操作 | MVP契約 |
| --- | --- |
| 編集 | 基本項目を更新する。 |
| 来週へ複製 | 新規予定として来週同時刻に複製する。元予定とは同期しない。 |
| 中止 | `status=cancelled` にする。物理削除はMVP対象外。 |

## 5. MVPデータ契約

### 5.1 状態

MVPの状態は3つだけとする。

| status | 意味 |
| --- | --- |
| `scheduled` | 予定済み。通常表示する。 |
| `cancelled` | 中止。変更点として表示し、通常の予定一覧では弱く表示する。 |
| `completed` | 完了。必要に応じて当日終了後に表示する。 |

`pending`、`approved`、`postponed`、`notified`、`confirmed` などの細かな状態はMVPに含めない。

### 5.2 種別

MVPの予定種別は4つだけとする。

| type | 意味 |
| --- | --- |
| `general` | 一般予定。 |
| `meeting` | 会議。 |
| `transport` | 送迎・移動。 |
| `support` | 利用者支援。 |

種別は表示補助と軽い絞り込みに使う。入力者に細かな分類判断を求めない。

## 6. 朝の予定サマリー

MVPでは「通知」という名称を使わず、「朝の予定サマリー」として扱う。

朝の予定サマリーに含めるものは以下に限定する。

| 項目 | 内容 |
| --- | --- |
| 今日の予定 | 当日の `scheduled` 予定。 |
| 変更された予定 | 前回確認後、または変更表示期間内の予定。 |
| 中止された予定 | 当日の `cancelled` 予定。 |

MVPからは以下を除外する。

- メール通知
- Teams通知
- プッシュ通知
- 個別リマインダー設定
- 利用者別または職員別の通知設定

## 7. 変更あり表示

### 7.1 判定条件

MVPでは、予定作成後に次のいずれかが変更された場合に「変更あり」と表示する。

| 対象項目 |
| --- |
| date |
| start |
| end |
| assignee |
| location |

title、target、memoの変更はMVPでは変更あり表示の必須条件にしない。必要になった場合は後続で拡張する。

### 7.2 表示期限

初期実装では、変更後24時間だけ「変更あり」を表示する。

確認済み状態はMVPには含めない。確認済み管理は、運用上必要と分かった時点でTo-Be側の通知・確認設計として追加する。

## 8. MVPから除外するもの

以下は初期実装には含めない。

| 除外項目 | 理由 |
| --- | --- |
| 高度な定期予定 | 例外処理が多く、初期実装を複雑化する。 |
| メール・Teams・プッシュ通知 | 設定、送信失敗、問い合わせ対応が増える。 |
| ISP期限管理 | 通常予定ではなく期限管理として分離する。 |
| 会議議事録との双方向連携 | 会議ドメインの設計が必要。 |
| 送迎システムとの自動同期 | 同期失敗、競合、外部契約が必要。 |
| 細かな承認状態 | 現場入力時の判断を増やす。 |
| 利用者別カスタム表示 | 個別設定と保守負荷が増える。 |
| 管理者向け設定画面 | 日常利用の目的から外れる。 |

## 9. To-Be設計との関係

`docs/design/schedules-to-be-design.md` は将来構想を定義する。Simple MVPはその全体像のうち、最初に現場へ出す範囲だけを切り出す。

| 文書 | 役割 |
| --- | --- |
| `docs/reports/schedule-feature-analysis-2026-07-13.md` | 現状と問題点を整理するAs-Is分析。 |
| `docs/design/schedules-to-be-design.md` | 将来の全体像と拡張設計。 |
| `docs/design/schedules-simple-mvp.md` | 初期実装の範囲と除外基準。 |

実装時に迷った場合は、本書のMVP判断基準を優先する。MVP基準を満たさない機能は、To-Be設計に存在していても初期実装へ入れない。

## 10. 受け入れ条件

Simple Schedule MVPは、以下を満たす状態を初期完成とする。

- 職員が今日の予定、担当、注意点、変更点を1画面で確認できる。
- 週表示で今週の予定を確認できる。
- 単発予定を最小項目で登録できる。
- 予定を編集、中止、来週へ複製できる。
- 状態は `scheduled` / `cancelled` / `completed` の3つに限定されている。
- 種別は `general` / `meeting` / `transport` / `support` の4つに限定されている。
- メール、Teams、プッシュ、個別通知、定期予定、外部同期は実装されていない。
