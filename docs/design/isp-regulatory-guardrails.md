# ISP 三層モデル — 制度運用ガードレール（AI エージェント向け）

> 最終更新: 2026-03-12  
> 対象: コード生成・設計判断を行う AI エージェント

---

## 絶対遵守ルール

### 1. 支援計画シート等は加算・減算の根拠文書である

支援計画シート等は、単なる支援設計文書ではなく、
**重度障害者支援加算、行動援護の未作成減算回避、行動障害支援連携加算の根拠文書** として扱うこと。

### 2. 追跡可能性の確保

支援計画シートおよび支援手順書兼記録には、少なくとも以下を追跡可能にすること。

- 作成者
- 作成者資格
- 作成日
- 対象サービス
- 対象加算
- 利用者の対象者判定情報
- 見直し日
- 実施記録
- 版管理
- 関係機関連携記録

### 3. 減算判定の月次管理

行動援護に関しては、支援計画シート等の未作成または作成業務不適切が
**月単位の減算要件** となるため、月次判定可能な状態管理と証跡保持を必須とする。

### 4. 作成者資格の検証

支援計画シートの作成者は名前だけでは不十分。
**資格要件を満たす人か** の判定情報が必須。

最低限の資格属性:
- 実践研修修了 / 基礎研修修了
- 行動援護従業者養成研修修了
- 中核的人材養成研修修了
- 修了日 / 有効な修了証確認有無

## 設計判断ガイドライン

### 三層は分離を維持する

ISP と支援計画シート、支援手順記録は **別リスト・別スキーマ** を維持。
統合は禁止。ただし参照（ID リンク）は必須。

### 加算ロジックは domain に閉じる

加算・減算の判定ロジックは `src/domain/billing/` に配置。
Repository やUI層に判定ロジックを書かない。

### 資格判定は staff-qualification ドメインに閉じる

「この作成者は要件を満たすか」の判定は `src/domain/staff-qualification/` に配置。
スタッフマスタの読み取りは data 層で行い、判定ロジックは domain で純粋関数として実装する。

### 対象者判定は user/eligibility に閉じる

「この利用者は重度障害者支援加算の対象か」の判定は利用者ドメインの判定関数で行う。
判定に必要な情報（障害支援区分・行動関連項目点数等）は Users_Master から取得。

## データ分類

| データ | 配置 | 理由 |
|---|---|---|
| ISP ステータス遷移 | `src/domain/isp/` | ドメインルール |
| 加算対象判定 | `src/domain/billing/` | 制度ルール |
| 減算リスク判定 | `src/domain/billing/` | 制度ルール |
| 作成者資格判定 | `src/domain/staff-qualification/` | 資格ルール |
| SP フィールド定義 | `src/sharepoint/fields/` | インフラ |
| SP ↔ Domain 変換 | `src/data/isp/sharepoint/mapper.ts` | データ変換 |
| OData クエリ | `src/data/isp/sharepoint/*Repository.ts` | インフラ |
| UI 表示ロジック | `src/features/*/` | プレゼンテーション |

---

## 禁止事項

1. ❌ ISP と支援計画シートを同一ドキュメント/スキーマに統合してはならない
2. ❌ 加算・減算の判定ロジックを UI コンポーネント内に書いてはならない
3. ❌ 作成者資格を単なる文字列で保存してはならない（構造化した判定可能な形式を使うこと）
4. ❌ 版管理を上書き方式で実装してはならない（旧版の保持が必須）
5. ❌ 見直し期限を省略してはならない（制度上6ヶ月以内のモニタリングが義務）
