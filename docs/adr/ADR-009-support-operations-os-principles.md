# ADR-009: Support Operations OS 設計原則の採用

> **Status**: Accepted  
> **Date**: 2026-03-16  
> **Related**: [principles.md](../product/principles.md), [ui-conventions.md](../product/ui-conventions.md), [roadmap.md](../product/roadmap.md)

---

## コンテキスト

Audit Management System は、障害福祉サービスの記録・管理システムとして開発が開始された。
開発が進むにつれ、以下のことが明らかになった。

1. **記録の蓄積だけでは価値が不足する** — 記録を入れても見直しや改善に自然につながらない
2. **分析を前面に出すと現場が離れる** — AI 提案を目立たせると現場職員が「使わされる」感覚を持つ
3. **ソースごとに提案 UI が異なると UX が崩壊する** — 申し送り・ABC・モニタリングそれぞれが別の提案形式を持つ
4. **判断履歴が残らないと監査・教育に使えない** — 採用/却下の記録なしに組織知化が進まない
5. **機能追加のたびに設計思想がばらつく** — 共通原則がないと各画面が独自解釈で進化する

この状況に対し、プロダクト全体を貫く設計原則が必要と判断した。

## 決定

**Support Operations OS 設計原則 10 箇条** をプロダクト憲章として採用する。

### 10 原則の要約

| # | 原則 | 一言 |
|---|------|------|
| 1 | 現場導線主体 | 現場導線を主役にする |
| 2 | 入力最小化 | 入力負担を増やさない |
| 3 | 候補提示 | 提案は候補として出す |
| 4 | 根拠付与 | 提案には根拠を付ける |
| 5 | 人間決定 | 最終決定は人が行う |
| 6 | 改善接続 | 記録を改善へつなぐ |
| 7 | 日次優先 | まず日次価値を作る |
| 8 | 組織知化 | 履歴を組織知に変える |
| 9 | 一貫性 | 一貫した OS として育てる |
| 10 | OS 定義 | 判断補助 OS として定義する |

## 理由

### なぜ「10 箇条」という形式か

- **チェック可能** — レビュー時に「原則 N に適合しているか」と確認できる
- **記憶可能** — 10 項目であれば全体を把握できる
- **拡張可能** — 将来的に原則を追加・修正できる構造
- **独立** — 各原則が独立しており、段階的に適用できる

### なぜ今このタイミングか

- 三ブリッジ（Assessment → Planning → Record → Monitor）が完成し、PDCA サイクル全体が実装された
- 提案統合レイヤー（ADR-010）で初めて「AI 提案を人が判断する」UI が必要になった
- Evidence 分析基盤（Phase 3-A/3-B/3-C）が整い、根拠追跡の仕組みが定着した
- ここで共通原則を定めないと、Phase 4 以降で各機能の設計思想がばらつく危険がある

### なぜ「判断補助 OS」という定義か

```
記録システム         → 記録を集めるだけ
ダッシュボード       → 表示するだけ
AI 自律決定システム  → 人の判断を排除する

判断補助 OS          → 記録 + 分析 + 候補提示 + 人が決定 + 履歴保存
                       の全ループを内包する
```

福祉業務において AI が自律的に意思決定することは、制度・倫理の両面から不適切である。
一方で、支援者が一人で全データを把握し続けることも現実的ではない。

「判断補助 OS」は、この両極の間の最適解として位置づけられる。

## 利点

- **プロダクトの一貫性** — 新機能が同じ思想で設計される
- **オンボーディング** — 新しい開発者が設計思想を理解しやすい
- **レビュー基準** — PR レビューで「原則 N」を引用して議論できる
- **OSS 展開準備** — 外部にも思想を説明できる文書が整う
- **監査対応** — なぜこの設計にしたかの根拠が残る

## トレードオフ

| 懸念 | 対応 |
|------|------|
| 原則に縛られすぎて柔軟性を失う | 原則は「ガイドライン」であり、合理的な理由があれば逸脱可能。その場合は ADR に記録する |
| 形骸化のリスク | 新機能の設計レビュー時に 10 箇条チェックを義務化する |
| 原則同士の矛盾 | 「入力最小化」と「根拠付与」は矛盾しうるが、「分析のために現場を苦しめない」で解決する（根拠は自動収集でカバー） |

## 適用方法

### 新機能設計時

1. [principles.md](../product/principles.md) の 10 原則を確認する
2. 該当する原則を設計ドキュメントに記載する
3. [ui-conventions.md](../product/ui-conventions.md) の共通 UI 規約に従う
4. [roadmap.md](../product/roadmap.md) のどのフェーズに位置するか記載する

### PR レビュー時

以下の観点でチェックする。

- [ ] 共通の `<ProposalCard>` を使用しているか（原則 9）
- [ ] 提案には「提案 + 根拠 + 参照元」の 3 点セットがあるか（原則 4）
- [ ] AI 提案に必ず人の承認ステップがあるか（原則 5）
- [ ] 採用/却下の履歴が保存されるか（原則 8）
- [ ] 用語が統一されているか（原則 9）

### 2026-04-17: ナビゲーションおよびルーター整合性の硬化 (PR #1510)

- **決定**: 8グループ構成のナビゲーション（Today, Records, Planning, Severe, Operations, Billing, Master, Platform）とルーター定義（`APP_ROUTE_PATHS`）を完全同期させ、診断ツール（`ORPHAN_ALLOWLIST`）を現行の Hub ベース設計に適合させた。
- **理由**: Hub 遷移化（詳細機能はハブ内に集約）という設計変更に対し、テストや診断ツールが古いサイドバー前提のままであった不整合を解消し、「一貫した OS」としての整合性（原則 9）を担保するため。
- **影響**: 未定義ルートやデッドリンクが診断ツールで自動検知可能となり、今後の機能追加時のデグレ耐性が向上した。
