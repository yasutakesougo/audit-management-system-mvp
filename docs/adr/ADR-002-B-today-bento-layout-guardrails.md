# ADR-002-B: TodayBentoLayout 設計・実装原則（ADR-002 追補版）

## Status
Accepted — 2026-03-25 (Supplement to ADR-002)

## Context
ADR-002は `/today` のデータフェッチと集約ロジック境界（`useTodaySummary`）を定義し、Dashboard（Decision Layer）への後戻りを防ぐバックエンド的制約を定めた。
本稿（ADR-002-B）は、その思想を**UIコンポーネントと空間レイアウトの次元へ拡張**したものである。
`/today` のフロントエンド実装である `TodayBentoLayout` が、画面の肥大化・情報の墓場化によって「情報を並べて考えさせるダッシュボード」に退行することを防ぐため、空間配置とUI挙動における厳格な不変条件（Guardrails）を定義する。

---

## Execution Layer Guardrails
### Design Concept / UI Pattern

## 1. ページの本質と役割

**Today は、通常の「情報を並べて考えさせるダッシュボード（Decision Layer）」ではない。**
Today は、**現場を前に進めるための操作起点画面（Execution Layer）** である。

このページの目的は、利用者に情報を比較・検討させることではなく、
**「今やるべきことに最短で着手させること」** にある。

### 基本原則（UX Principle）

* **探させない**
  最優先行動への視線誘導を一本化し、最初の数秒で迷わせない。

* **考えさせない**
  状態比較のための情報提示ではなく、「次に進むべき移動先」を即時提示する。

* **戻らせない**
  実行対象を常時表示し、中断後でもすぐに作業を再開できるようにする。

### 行動順に沿ったUI構造

Today は「認知 → 判断 → 実行」という一般的な情報画面の流れではなく、
**現場の行動順** に合わせて空間配置を行う。

**[即実行] → [状況把握] → [対象処理] → [必要時参照]**

上にあるほど即時行動に近く、下に行くほど補助情報になる。
この空間的な優先度設計を崩してはならない。

---

## 2. アーキテクチャと4つの空間レイヤー

### L0: Global Alert Layer
#### 役割
ページ全体の前提を壊す例外・逸脱状態を、最上段で通知する。
#### 配置
* `TodayExceptionAlerts`
#### 原則
* 「通常運転が崩れている」ことを、Hero より前に伝える。
* ただし、常時赤アラートだらけにしてはならない。
* Alert は重要だが、**Hero の主導権を奪ってはならない**。
* 表示条件は厳格に制御し、ノイズ化を防ぐ。

---

### L1: Primary Action Layer
#### 最初の数秒でユーザーを動かす層
#### 役割
最初の数秒でユーザーに行動を起こさせる、単一の司令塔とする。
#### 配置
* `HeroActionCard`（ZONE A）
#### 原則
* Hero は **常に「今やるべき1件」だけ** を示す。
* `sceneAction`（場面全体の優先事項）を最優先とし、なければ `nextAction` にフォールバックする。
* Hero は **single-threaded entry point** として機能しなければならない。
* 複数CTA、補助説明、2番目の優先事項の追加は禁止する。

---

### L2: Operational Overview Layer
#### 実行中の状況把握を支える層
#### 役割
実行の合間に、
* どこが詰まっているか
* 次にどこへ回るべきか
* 本日の進捗がどうなっているか
を軽量に把握させる。
#### 配置
* `ProgressRings`
* `ScheduleOpsHighLoadTile`
* （ZONE B）
#### 原則
* 単なる閲覧用メトリクスにしてはならない。
* ここは「見るための数字」ではなく、**次の対象一覧・処理へ進むナビゲーション導線** である。
* 押せば必ず、対象画面・対象一覧・次の行動へ遷移できること。
* ダッシュボード化しすぎず、実行支援のための軽量オーバービューに留める。

---

### L3: Execution Workspace Layer
#### 実務対象と補助情報を扱う層
この層は、ページ下部が「情報の墓場」になることを防ぐため、
**C1（常時表示）と C2（必要時参照）に厳格に分離する。**

### C1: 常時表示領域
#### 今日必ず触るものだけを置く
#### 役割
そのまま行動に移れる、実務対象のリスト群。
別画面から戻ってきた際にも、再開しやすさを担保する。
#### 配置
* `UserCompactList`
* `HandoffPanel`
* `CallLogSummaryCard`
#### 原則
* ここは「便利なもの置き場」ではない。
* **現場が今日必ず触るものだけ** を厳格に置く。
* 中断復帰後の再開性を最優先する。
* 追加候補は「毎日高頻度で触るか」「ここに常駐していないと再開性が落ちるか」で審査する。

### C2: 必要時参照領域
#### 補助情報は折りたたんで持つ
#### 役割
運用上は重要だが、常時開いておく必要はない補助情報を保持する。
#### 配置
* `TodayServiceStructureCard`
* `PlanningWorkflowCard`
* `TransportStatusCard`
#### 原則
* **デフォルト閉で成立するものだけ** を置く。
* 開かないと仕事にならない情報は、C2に置いてはならない。
* その場合は C1 への昇格を検討する。
* 管理情報・参照情報・補足情報は、Execution Layer を圧迫しないよう C2 に閉じ込める。

---

## 3. 守るべき不変条件 (Anti-Patterns / Guardrails)

以下に該当する改修は、
**Execution Layer を Decision Layer に劣化させる退行変更** とみなし、
Pull Request レビューで原則 Reject する。

### 1. Hero の肥大化を許さない
* 「ついでにこれも表示したい」
* 「2番目の優先タスクも見せたい」
* 「説明文を少し足したい」
これらはすべて Hero の一本化を壊す。Hero は **最優先タスクを一つだけ提示する領域** である。

### 2. Progress のメトリクス化を許さない
* 押しても何も起きない数字
* 見るだけのグラフ
* 行動導線を持たない進捗表示
これらは Today を再び「眺める画面」に戻してしまう。Progress は **ナビゲーション可能な行動導線** でなければならない。

### 3. C1 の飽和を許さない
* 「便利そうだから置く」
* 「あった方が見やすいから追加する」
* 「管理者は使うかもしれないから常時出す」
この種の追加は C1 をすぐ飽和させる。C1 は **今日の実務に必須なものだけ** に限定する。

### 4. 管理情報の侵食を許さない
* 計画管理
* 体制確認
* 監査・分析
* 中長期の見直し導線
これらが前面に出すぎると、Today は Execution Layer ではなくなる。管理業務は Dashboard / Planning / Admin 側に逃がし、Today の純度を守る。

### 5. Alert による Hero の埋没を許さない
Alert は Hero より上に置くが、常時大量表示して Hero の主導権を奪ってはならない。例外通知は **強いが少ない** 状態を維持する。

---

## 4. 実装判断のためのレビュー基準 (PR Review Checklist)

PRレビュー時は、機能の有無ではなく、次の観点で判断する。

- [ ] この変更は、最初の5秒の行動開始を速くするか
- [ ] Hero の一本化を壊していないか
- [ ] Progress を「見るだけ」にしていないか
- [ ] C1 の常時表示領域を膨張させていないか
- [ ] C2 に逃がすべき情報を前面に出していないか
- [ ] 追加情報が Execution Layer ではなく Decision Layer 的になっていないか
- [ ] 中断復帰後の再開しやすさを損なっていないか

---

## 5. 一文での定義

> TodayBentoLayout は、当日の最優先行動を提示し、進捗の詰まりを可視化し、実行対象へ最短で到達させるための Execution Layer である。上部ほど即時行動、下部ほど補助情報となるよう情報優先度を空間配置し、常時表示対象と必要時参照対象を明確に分離する。
