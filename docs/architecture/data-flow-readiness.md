- **関連ドキュメント**: `docs/architecture/frontend-structure.md`（構造レポート）, `docs/architecture/feature-inventory.md`（機能棚卸）

## 第1章 調査の目的と判定基準

フロントエンドの構造把握および機能の棚卸を経て、本調査では「各主要機能が実際に本番データ（SharePoint）やデモデータ（InMemory / LocalStorage）とどのように連携し、実運用（現場投入）に耐えうる状態（レディネス）にあるか」を判定する。

### 1.1 評価の5つの観点

単に「データを保存できるか」だけでなく、現場での操作ミスや一時的な通信切断などの実運用シナリオに耐えうるかという観点から、以下の5つを評価軸とした。

| 観点 | 評価・確認内容 |
|---|---|
| **画面導線と主要操作** | URL、ナビゲーション、主要操作ボタン（新規作成、修正、保存）が整合しているか |
| **保存と再取得（戻せるか）** | 新規保存（Create）後にデータが再取得でき、画面再読み込み後も正しく再表示・再現されるか |
| **堅牢性と二重送信防止** | 保存処理中の送信ボタンの disabled 制御や、ローディング表示などがあるか |
| **訂正・再保存の境界** | 既存データの修正（Update / Correct）と新規作成の境界がコード上で明確に区別されているか |
| **Repository分離とエラー処理** | UIが直接APIを叩いておらずリポジトリ境界が分離されているか、失敗時のエラー表示や代替保存（LocalStorage）があるか |
| **テストファイル数** | ドメイン内のテストファイル（`__tests__`配下など）の正確な合計数 |

### 1.2 実運用レディネスランク定義

本レポートでは、実運用に向けた状態を以下の5段階（A〜E）で定義する。

| ランク | 判定定義 | 移行要件・次アクション |
|---|---|---|
| **A: 運用可能** | SharePointまたは代替永続化層まで保存・取得・再表示でき、主要テストおよびエラー・二重送信制御がある。 | 即時本番/現場試験投入可能 |
| **B: 試験運用可** | 主要フローは動作するが、一部がInMemory/mock、またはテストや例外処理、UI制御が一部不足している。 | テスト追加、UI制御の微修正 |
| **C: 部分実装** | UI / hook / repository の一部は存在するが、永続化フローが未完成、またはアーキテクチャ分離ルールに違反している。 | 境界分離、保存・更新フローの再構築 |
| **D: 導線のみ** | 画面導線やプレースホルダーが中心で、実際のデータ保存やAPI接続が未実装。 | データ連携層の新規実装 |
| **E: 要再調査** | リポジトリ境界が不明瞭、古いレガシー実装が混在し、動作に不安定さがあるため再設計・再調査が必要。 | 境界設計のやり直し、依存整理 |

## 第2章 主要ドメインの実運用レディネス分析

実運用への投入が近い主要ドメイン、および監査システムとして中核となる機能領域について精査した。

### 2.1 Kiosk 領域（現場端末・トイレ記録・手順フロー）

現場でスタッフや利用者が操作するキオスク端末は、トイレ記録の入力および支援手順の進捗管理を行う。

#### データフローと実装特徴
- **トイレ記録 (Toilet)**:
  `IToiletRecordRepository`（インターフェース）を通じて操作を行う。`toiletRepositoryFactory` により、デモ/オフライン時は `LocalStorageToiletRecordRepository` を選択し、本番時は `SharePointToiletRecordRepository` に切り替える。
- **支援手順 (Procedures)**:
  `useDailyProcedureFlowPreview` フックが `daily` ドメインの Zustand store （`useExecutionStore` / `useProcedureStore`）からリアクティブにデータを結合し、表示用の手順ステップに変換する。

#### レディネス評価

- **画面導線と主要操作**: `/kiosk/toilet` (トイレ入力用グリッド) と `/kiosk/users/:userId/procedures` (手順一覧) が完備されている。
- **保存と再取得**: トイレ記録は保存後即座に `refresh` が走り、画面再読み込み時も LocalStorage または SharePoint から再取得・再表示される。
- **二重送信防止**: トイレ記録の打刻ボタンには、打刻中（busy）や再試行時のローディング状態制御が備わっている。
- **訂正・再保存の境界**: トイレ記録の更新は `IToiletRecordRepository.update` (Correct patch) として明示的に定義されており、新規打刻と明確にインターフェースが分離されている。
- **Repository分離とエラー処理**: `toiletRepositoryFactory.ts` により、UIはリポジトリ境界にのみ依存している。SharePointへの接続失敗時は `LocalStorage` が代替永続化層として機能する。
- **テストファイル数**: **14 ファイル**。

- **実運用レディネスランク**: **A**
- **特記事項**: LocalStorage を用いた代替永続化と、SharePoint 連携の自動フォールバックが秀逸であり、ネットワークが不安定な現場端末としてのレディネスを満たしている。

---

### 2.2 Daily / Record Quality 領域（支援記録とレビュー）

日々の支援記録の入力および、その入力内容に対する監査品質レビューを行うコア領域。

#### データフローと実装特徴
- `DataProviderDailyRecordRepository` から SharePoint へ保存する。
- 保存完了のトランザクションに紐付き、アプリケーション層のユースケース `saveDailyRecordWithQualityReview` が `RecordQualityReview` リポジトリを呼び出し、品質レビューメタデータを生成する。

#### レディネス評価

- **画面導線と主要操作**: `/daily/activity` (活動入力) や `/records/quality-review` (品質判定画面) など、必要な全UIと操作ボタンが存在する。
- **保存と再取得**: 保存後にデータは SharePoint (本番) または InMemory (デモ) へ永続化され、画面再読み込み後も一覧やフォームに復元される。
- **二重送信防止**: `DailyRecordPage` 等の送信処理では送信中の loading 状態およびボタンの disabled 制御が施されている。
- **訂正・再保存の境界**: 新規作成と既存修正が `save` メソッドに渡される mutationParams の有無やキー（date/userId）の判定によって区別されている。
- **Repository分離とエラー処理**: `repositoryFactory.ts` により、UIはドメインリポジトリ境界にしか依存しない。エラーハンドリングはユースケース層で統合管理されている。
- **テストファイル数**: **77 ファイル**（`daily`: 72, `record-quality`: 5）。

- **実運用レディネスランク**: **A**
- **特記事項**: Ports & Adapters が非常に高いレベルで適用されており、レビューとの連携ユースケースに対するテストも網羅的で実用性が高い。

---

### 2.3 Users 領域（利用者マスタ）

全機能の土台となる利用者マスタの管理。

#### データフローと実装特徴
- `useUsers` / `useUsersQuery` と `useUserForm` フックから `getUserRepository()` を呼び出す。
- 本番リポジトリ `DataProviderUserRepository` は `resolveProvider()` を介してグローバルなデータプロバイダ（キャッシュ・REST API）と接続している。

#### レディネス評価

- **画面導線と主要操作**: `/users` (利用者一覧・追加フォーム) と `/users/:userId` (詳細情報・更新) が実装されている。
- **保存と再取得**: `UserForm` で保存後、キャッシュおよび SharePoint が更新され、再読み込み後も正しく再表示される。
- **二重送信防止**: フォーム送信中の `busy` 状態制御が UI 側（`UserForm.tsx`）に実装されている。
- **訂正・再保存の境界**: 新規登録と既存更新でフォームの挙動および API 呼び出し（Create/Update）が明確に分かれている。
- **Repository分離とエラー処理**: `UserRepository` インターフェースによりクリーンに分離されている。プロバイダ解決失敗時のフォールバック処理も定義されている。
- **テストファイル数**: **21 ファイル**。

- **実運用レディネスランク**: **A**
- **特記事項**: 実装およびテスト体制ともに安定しており、マスタデータ管理システムとしての動作に懸念はない。

---

### 2.4 Attendance 領域（通所管理・出欠記録）

利用者の入退所（通所）打刻および送迎方法の変更を扱う領域。

#### データフローと実装特徴
- `useAttendance` を介して、`AttendanceRepository` からデータを取得・更新する。
- 職員が打刻した結果は、Zustand store (`store.ts`) にてリアクティブに保持され、同一画面内の他のコンポーネント（通所状況カード等）へ反映される。

#### レディネス評価

- **画面導線と主要操作**: `/daily/attendance` (通所打刻ボード) が完備されている。
- **保存と再取得**: 打刻後は即時ストアと SharePoint が更新され、再読み込み後も打刻時間が正しく表示される。
- **二重送信防止**: 打刻ボタンのアクション中に多重打刻が防止される制御がロジック層・UI層双方に組み込まれている。
- **訂正・再保存の境界**: 登所（入所）と退所の打刻ステータス遷移が `attendance.logic.ts` で厳格にガードされており、誤打刻や状態の上書きを防ぐ設計になっている。
- **Repository分離とエラー処理**: ファクトリ経由で分離され、自動退所処理 (`attendance.autoNext.ts`) などのエラーも適切に処理される。
- **テストファイル数**: **14 ファイル**。

- **実運用レディネスランク**: **A**
- **特記事項**: ビジネスルール（入退所の順序関係、送迎方法の整合性）が独立したロジックファイルに抽出され、単体テストで保証されている。

---

### 2.5 Checklist / Exception Center 領域

自己点検ルールの管理と、データ未入力・異常値の横断監視を行う。

#### データフローと実装特徴
- **Checklist (自己点検)**:
  `useChecklistApi` は、`lib/spClient` からインポートした `useSP` クライアントを**直接呼び出し**て、SharePoint API と直接通信している。
- **Exception Center**:
  `useExceptionDataSources` フックが、他ドメイン（Daily, Handoff, Users, SupportPlan）のリポジトリを複合的に呼び出し、データを集約した上で例外検出エンジンに渡す。

#### レディネス評価

- **画面導線と主要操作**: `/checklist` (チェックリスト一覧・追加) と `/admin/exception-center` (例外センター) が存在する。
- **保存と再取得**:
  - Checklist: 新規追加後にリストが更新され再表示されるが、SharePoint が無いデモ環境では動作しない。
  - Exception Center: 読み取り専用。他ドメインの更新を検知して自動で再取得 (`refetch`) する口を有する。
- **二重送信防止**: Checklistには追加中 (`busy`) ガードが実装されている。
- **訂正・再保存 & Repository分離**:
  - Checklist: **C（境界分離なし）**。リポジトリファクトリが適用されておらず、InMemory用のデモ実装がない。
  - Exception Center: **B（一部結合）**。データソース自体は他ドメインのリポジトリ境界を再利用しているが、例外センター専用の永続化層（修正アクションなど）の境界分離は薄い。
- **テストファイル数**:
  - Checklist: **1 ファイル** (`checklistLogic.spec.ts` のみ。UIやAPIのテストは **0件**)。
  - Exception Center: **21 ファイル** (他ドメインとの連携テストを含む)。

- **実運用レディネスランク**:
  - Exception Center: **B**
  - Checklist: **C**
- **特記事項**: Checklist 領域は、現在のアーキテクチャ規律（`repositoryFactory` による切り替え）を満たしておらず、ローカル/デモ環境で機能がエラーになる問題がある。

## 第3章 永続化データソースと境界分離の評価

本システムでは、API接続とUIの間に「リポジトリ境界」を設けるか否かで、テスト容易性とデモ環境での動作品質に大きな差が出ている。

### 3.1 境界分離と直接参照の比較

#### パターンA: 境界分離型（推奨・標準パターン）
UI コンポーネントおよび画面特有の Hook は、抽象的なリポジトリインターフェースに依存する。本番（SharePoint）とデモ（InMemory）はファクトリで切り替える。
* **適用ドメイン**: `daily`, `users`, `attendance`, `schedules` など主要ドメインのほぼ全て。
* **メリット**: 通信エラー時のモック検証や、オフライン時の LocalStorage 代替保存が容易。

#### パターンB: API直接参照型（要改善パターン）
UI や API 連携フックが、共通 API クライアント（`useSP`）を直接呼び出してクエリを発行する。
* **適用ドメイン**: `compliance-checklist` (Checklist)
* **デメリット**: 開発・デモモードでも本番の SharePoint リストが存在しないとクラッシュする。単体テストが困難。

## 第4章 次に実装・補強すべき改善候補の提案

実運用（現場導入）に向けて、技術的負債の解消と品質向上のための改善アクションを以下に提案する。

### 提案1: Checklist 領域の `repositoryFactory` 適用（優先度：高）
- **課題**: `compliance-checklist` 領域が `useSP` に直接依存しており、デモ動作や結合テストが行えない。
- **対策**: `IChecklistRepository` を定義し、`DataProviderChecklistRepository` と `InMemoryChecklistRepository` をファクトリで切り替える構造にリファクタリングする。
- **効果**: デモ環境での動作安定化と、API/UI層のモックテスト追加が可能になる。

### 提案2: Kiosk 領域でのオフライン同期エンジンの実装（優先度：中）
- **課題**: `LocalStorageToiletRecordRepository` による一時退避機能はあるが、オンライン復帰時に SharePoint 側へプッシュする同期トリガーが未実装。
- **対策**: ネットワークステータス (`navigator.onLine`) の変化を検知し、未同期ログを自動またはボタン操作でプッシュする同期コンポーネントを導入する。
- **効果**: 現場での Wi-Fi 途絶に対する堅牢性が完成し、実運用端末としての安心感が向上する。

### 提案3: 二重送信防止のコンポーネントレベルでの共通ガード（優先度：低）
- **課題**: 各ドメインのUIで個別に `busy` 状態をステート管理してボタンを disable にしているため、実装漏れのリスクがある。
- **対策**: 送信処理（Promise）をラップして自動的に loading/disabled を制御する共通ボタンコンポーネントやカスタムフックの導入。
- **効果**: 開発者の実装負荷を軽減し、全フォームでの二重送信トラブルを防止する。
