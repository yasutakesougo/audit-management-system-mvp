# Navigation Philosophy (福祉OSのナビゲーション哲学)

本ドキュメントは、監査・業務管理統合システム（福祉OS）のナビゲーション設計における「思想」と「制約」を定義するものです。
UIを追加・変更する際は、必ずこの思想に照らし合わせて判断してください。

## 1. Core Philosophy: "Today" is the Primary Driver

**「サイドバーから探す」のではなく「Todayからアクションへ飛ぶ」**

当システムにおいて、サイドメニューは Primary Navigation ではありません。
ユーザーの行動を牽引するのは **「Today（行動の司令塔）」** 画面が発火する Signal です。
サイドバーはあくまで**「Secondary Index（辞書的な裏口）」**であり、探索的・回遊的なUIを持たせるべきではありません。

## 2. No "Icon Soup" (汎用アイコンの乱用禁止)

現在、サイドメニューの各項目には意図的にアイコン（`icon: undefined`）を設定していません。
これは以下のB2B業務システム特有の課題を回避するためです。

- **認知負荷の増大**: 20〜30以上の機能に汎用アイコン（📄、⚙️、📁など）を付与すると、意味が重複・飽和し、ユーザーが「アイコン解釈」に脳のメモリを消費してしまいます。
- **スキャン速度の低下**: 直感的な判別がつかないアイコンの羅列は、明確な「テキスト（機能名）」のスキャンを阻害します。

## 3. Purpose-Driven Grouping (目的別のグループ化)

ナビゲーションは「機能の類似性」ではなく、**「ユーザーの業務目的」** を軸にグループ化されています。

- `daily`: 現場の実行（今日の業務、記録、申し送り）
- `assessment`: 支援計画・アセスメント（ISP作成、評価、分析）
- `record`: 記録・振り返り（月次、実績、運営状況）
- `ops`: 拠点運営（請求、勤怠）
- `admin`: マスタ・管理設定

ユーザーは「自分の今の目的（グループ見出し）」→「具体的なテキスト」の順で視線を動かすため、アイコン以上の強力な道標となります。

## 4. Known Limitations and Future Mitigation

### 課題: Collapsed Sidebar (折りたたみ時の視認性)

全項目からアイコンを削ぎ落とした結果、サイドメニューを折りたたんだ際に何もない（ホバーでツールチップ表示のみ）状態となっています。

### 今後の解決方針

もし現場から「画面を広く使いたいが、主要機能だけは素早く切り替えたい」という強い要望が出た場合は、全項目にアイコンを付けるのではなく、以下のいずれかのアプローチを取ります。

1. **Top-Level Only**: `Today`, `日次記録`, `スケジュール` など、極めて利用頻度が高い3〜5項目のみ例外的にアイコンを付与する。
2. **Group Icons**: 項目ごとではなく、`daily` や `ops` などの「グループ」単位で親アイコンを見せる構造へ移行する。

## 5. Ultimate Goal: Command Palette (Cmd + K)

ナビゲーションの最終形態として、階層を辿る行為自体をスキップする **Command Palette（メニュー検索ボックス / フルスクリーン検索）** の導入を目指しています。
これにより、ユーザーは考えることなく最短距離で目的の画面や対象者ページへ到達できるようになります。
