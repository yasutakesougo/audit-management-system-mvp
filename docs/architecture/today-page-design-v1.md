# Today画面 情報設計 v1 (福祉OSのホーム)

## 1. コンセプト

**「Today は各画面の寄せ集めではなく、各モジュールが発火する Signal を統合して、役割別に行動へ変換する層である。」**

Today画面は単なる「システムトップ」でも「メニューの再掲」でもなく、**「福祉OSのコックピット（行動の司令塔）」** として機能する。
利用者がシステムを開いて最初に見るべきは機能一覧ではなく、**「今日、自分が行うべき具体的なアクション」** と **「支援する上で知っておくべきアラート」** である。

システムにある「記録」「計画」「分析」の各モジュールから、**「今日必要な断面（Signal）」** のみを抽出し、優先順位をつけて提示する。表示されたカードをクリックすれば、そのまま対象の入力や詳細画面へ遷移し、迷わず処理を完了できる（One-Click to Action）。

---

## 2. 画面レイアウトと情報階層 (Layer)

画面は以下の4層（4 Layers）で情報を評価・配置し、現場の認知負荷を最小化する。

```mermaid
flowchart TD
  A[Today画面<br/>福祉OSのホーム] --> B[Layer 1: Hero / Next Action]
  A --> C[Layer 2: 今日の実行キュー]
  A --> D[Layer 3: 注意・連携・期限]
  A --> E[Layer 4: 深掘り導線]

  subgraph "P0: いますぐ対応"
    B --> B1[未記録]
    B --> B2[期限超過]
    B --> B3[承認待ち]
    B --> B4[本日の重要アラート]
  end

  subgraph "P1: 今日中に抜け漏れなく処理"
    C --> C1[日次記録の開始]
    C --> C2[健康記録の入力]
    C --> C3[申し送り確認]
  end

  subgraph "P2: 注意して見るべき利用者・予定"
    D --> D1[高リスク/体調注意 利用者]
    D --> D2[モニタリング期限 (近日)]
    D --> D3[ISP見直し候補]
    D --> D4[分析からの示唆 (環境変化など)]
  end

  subgraph "P3: 必要なら深掘り先へ遷移"
    E --> E1[>> 記録一覧へ]
    E --> E2[>> ダッシュボードへ]
  end
```

## 3. 表示優先順位 (Priority) と 表示ルール

Today画面が情報過多で崩壊するのを防ぐため、厳格な表示優先順位（Priority）を設ける。
**「終わったこと・正常なこと」は表示せず、「まだやっていないこと（未処理）」「異常や注意」を主語にする。**

| 優先度 | 定義                                          | 具体的なWidget・データソース例                                                                                           | アクション                                                |
| :----- | :-------------------------------------------- | :----------------------------------------------------------------------------------------------------------------------- | :-------------------------------------------------------- |
| **P0** | **すぐ行動が必要<br/>（業務ブロック要因）**   | ・期限超過のモニタリング<br/>・昨日/本日の未入力記録<br/>・未読のままの重要申し送り<br/>・承認待ちの記録（サビ管の場合） | 「XXを記録する」「承認画面へ」                            |
| **P1** | **今日中に処理<br/>（ルーチン業務の起点）**   | ・本日のスケジュール/支援予定<br/>・健康記録のショートカット<br/>・朝礼の申し送り確認                                    | 「予定を見る」「健康記録を入力」                          |
| **P2** | **見ておくと良い<br/>（質の高い支援のため）** | ・分析アラート（例「行動変化あり」）<br/>・ISP更新候補（期限まで2週間など）<br/>・スタッフ向け全体通知                   | 「利用者の直近記録を見る」<br/>「アセスメント結果を確認」 |
| **P3** | **深掘り導線<br/>（一覧・集計・マスタ）**     | ・過去の記録一覧へのリンク<br/>・拠点全体の分析ダッシュボード<br/>・利用者マスタ                                         | 「詳しく見る (遷移)」<br/>※Todayには直接一覧を出さない    |

## 4. 役割別（Audience）の見え方

メニューナビゲーション（`navAudience`）の設計思想を引き継ぎ、ログインユーザーの役割に応じてToday画面のWidgetの重み付けを動的に変更する。

### 一般スタッフ（現場職員）

- **最優先**: 「今日の支援予定」「未記録の解消」「申し送りの確認」
- **注意**: 「担当利用者の体調・配慮事項」
- **不要・縮小**: 拠点全体の進捗や、自分が関わらない更新候補

### サービス管理責任者（サビ管） / 管理者

- **最優先**: 「モニタリング期限の管理」「ISP更新候補」「記録の承認」
- **注意**: 「高リスク利用者の全体像」「担当者未定の業務」
- **別枠**: 現場の記録ショートカットより、全体の進捗・アラートを上位に置く。

## 5. UI実装アーキテクチャ

既存モジュールからの依存をクリーンに保つため、Today画面は各モジュールから **Domain Signal（アラートやサマリーのデータ）** を受け取って表示する「薄い統合層」として実装する。

### 構成要素と責務

1. **Domain: Today Signals**
   - 各Feature（例: `ibd/plans`, `daily/records`）から、`overdue`, `pending`, `risk` などの規格化されたSignalをToday向けに提供するインターフェース。
2. **Application: Action Queue Builder**
   - `buildTodayActionQueue()` 等で、集めたSignalをマージし、P0〜P3の優先順位付けとAudienceごとのフィルタリングを行う。
3. **UI State: `useTodayPageState`**
   - フィルタ（担当のみ/全体）や、完了したタスクの楽観的非表示（optimistic update）、ウィジェットの折りたたみ状態を管理する。
4. **Presentational: Widget群**
   - `NextActionPanel`（未完了タスク）
   - `DailyExecutionCards`（記録ショートカット）
   - `AlertsPanel`（注意・示唆）
   - 画面自体は共通骨格（3カラムまたはグリッド）を持ち、これらWidgetを配置する。

## 6. MVPスコープでの実装ブロック

まず最初に取り組むべきMVP（Minimum Viable Product）としてのToday画面は、以下の5ブロックで構成する。

1. **Hero Summary**: 「お疲れ様です。未記録がX件、未読の申し送りがY件あります。」
2. **Next Action Panel**: P0（すぐ行動が必要）なアクションのリスト（ボタン押下で直行）。
3. **今日の記録ショートカット**: P1（今日中に処理）の入口。日次記録・健康記録への大きなボタン。
4. **注意利用者 / 期限アラート**: P2（見ておくと良い）のリスト。期限が近いもの、リスクフラグが立ったもの。
5. **申し送り / 共有事項**: P1/P2。最新の連携事項。

これにより、「ここを見れば今日の業務が回る」という強い体験（現場OS感）を最速で構築する。
