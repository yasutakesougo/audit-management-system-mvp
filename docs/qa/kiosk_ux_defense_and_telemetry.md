# Kiosk UX 防衛と Telemetry 統合戦略

今回の検証で確立した **「現場で迷わない（探さない・必ず帰れる）UX基準」** を、今後の開発でリグレッション（退行）させないための具体的な仕組み作りに関する運用ガイドです。

---

## 1. UX Regression 自動テスト化（E2E）

手動テスト（PRチェックリスト）に加えて、CI/CDで確実にキオスクモードのナビゲーション崩壊を検知するPlaywrightの自動テストが組み込まれています。

### 自動テストで担保する観点
1. キオスクモードにおいて、サイドバーがDOMツリーから隠蔽（非表示）されていること。
2. Today画面から、主要5機能（スケジュール、支援記録、申し送り、議事録、お部屋管理）へのリンク要素が**可視状態**で存在し、1クリックで正しく遷移できること。
3. 遷移先（例：`/schedules/week` や `/room-management`）で、画面左上に「**← 今日の業務に戻る**」ボタンが存在し、押下するとToday（`/today`）に戻れること。
4. FAB（長押しメニュー）が表示されていること。

---

## 2. Telemetry を用いた導線エンゲージメント計測

「1タップでいける」状態を作った後は、**「現場の職員が本当にその最短ルートを使っているか（= 迷わず押しているか）」** を計測します。
`KioskQuickLinks` 等に配置されたボタンクリックに対して、`kiosk_ux_event` Telemetryイベントを発火させます。

### 推奨・実装済みのイベントスキーマ設計

| イベント名 | プロパティ（Property） | 説明 |
| :--- | :--- | :--- |
| `ux_navigate_from_today` | `source: 'today'` | Todayにあるクイックリンクから遷移したことを記録 |
| | `target: 'schedules'`等 | どこに向かったか |
| `ux_open_fab_menu` | `source: 'fab'` | FABを開いて遷移した（最短ルートが見つからなかった可能性） |
| `ux_return_to_today` | `source: 'header_back'` | 「今日の業務に戻る」を利用したか |

### なぜこれを計測するのか？（運用時の確認観点）
データの流れが始まった後、以下の**3つのKPI（観点）**だけを定期的にダッシュボードで確認します。

1. **`ux_navigate_from_today` の `target` 別件数**: どの機能が1タップで最頻利用されているか。
2. **`ux_open_fab_menu` の発生頻度**: この数値が全体の遷移数に対して異常に高い場合、「必要なクイックリンクが欠けている」「直感的に見つけづらい」というUX課題の早期発見に繋がります。
3. **`ux_return_to_today` の `source` 別件数**: どこからダッシュボードに戻ってくることが多いか。

👉 この3観点を見るだけで**「1タップ導線が機能し、FAB頼みの迷子状態が起きていないこと」**を完全観測できます。
