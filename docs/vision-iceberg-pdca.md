# なぜ氷山PDCAは普通の記録システムとレベルが違うのか

> このドキュメントは、クロノート Link の氷山PDCA機能が
> 既存の福祉記録システムと構造的にどう異なるかを整理したものです。

---

## 結論

クロノート Link は単なる「福祉記録システム」ではなく、
**支援の思考そのものをソフトウェア化している**。

その核が **氷山PDCA** であり、普通の記録システムとの違いは
**3つの構造の違い** にある。

---

## ① 「出来事記録」ではなく「原因分析」になっている

### 普通の記録システム

```
出来事 → 支援記録 → 保存
```

例：
> 15:30 外出時に大声。スタッフが声かけ。落ち着いた。

これは **出来事ログ** であり、そこで止まる。

### 氷山PDCA

```
出来事 → 原因分析 → 仮説 → 支援実験 → 評価
```

つまり **記録システム → 仮説検証システム** に変わっている。

これは福祉の世界では本来 **行動分析（ABA）** に近い構造であり、
表面の行動ではなく、その下にある感情・認知・環境・ニーズを
氷山モデルで構造的に分析する。

---

## ② 「支援改善サイクル」がデータ構造として実装されている

### 普通のシステム：データは分断されている

| 書類 | 作るタイミング | システム |
|------|-------------|---------|
| 支援記録 | 毎日 | 記録ソフト |
| モニタリング | 3ヶ月 | Excel or 紙 |
| 支援計画 | 半年 | 別の書類 |

3つが **別の書類・別のシステム** で、連動していない。

### 氷山PDCA：同じデータ構造で一気通貫

```
Daily Record（支援記録）
      ↓
Insight（気づき）
      ↓
PDCA（分析・改善サイクル）
      ↓
ISP（支援計画シート）
```

これはシステム設計的には **ナレッジ生成エンジン** である。

---

## ③ 支援の「成功パターン」を蓄積できる

### 普通の記録システムの問題

```
良い支援 → 記録される → 埋もれる → 次の職員が知らない
```

これは現場で日常的に起きている。

### 氷山PDCA

```
PLAN → DO → CHECK → ACT → 支援計画反映
```

**成功した支援が標準化される** 構造。
個人の知見がチームの知識に変換される。

---

## 既存の福祉システムとの比較

### 既存システムの3分類

| 類型 | 機能 | 例 |
|------|------|-----|
| ① 記録型 | 日々の支援記録入力 | 介護ソフト |
| ② 請求型 | 国保連への請求処理 | 請求ソフト |
| ③ 計画書型 | ISP（個別支援計画）の作成 | 帳票ソフト |

これらは **別システム** として存在し、データは連動しない。

### クロノート Link

```
支援記録 → 分析 → モニタリング → 計画更新 → 監査証跡
```

これを **一つの運用OS（Support Operations OS）** として統合。

---

## 本質的な価値：福祉 × ナレッジマネジメント

```
記録 → データ → 知識 → 支援改善
```

これは企業でいうと **トヨタのカイゼン** と同じ構造：

```
現場観察 → 問題発見 → 小さな実験 → 標準化
```

それを **福祉支援に持ち込んでいる** のが氷山PDCA。

---

## 進化の入り口：AI分析への接続

氷山PDCAのデータ構造は、将来のAI分析と自然に接続できる。

```
支援記録データ → パターン発見 → 支援提案
```

例：
> 利用者A・B・Cで「午後外出前」に不安行動が5回発生
> → 「午後外出前の見通し不足が原因の可能性」をAIが提案

氷山PDCAが **AIが学習できる構造** になっている点が重要。

---

## まとめ

| 観点 | 普通の記録システム | クロノート Link |
|------|-----------------|---------------|
| 記録の性質 | 出来事ログ | 原因分析 |
| データ構造 | 分断（別書類） | 統合（一気通貫） |
| 知識の扱い | 埋もれる | 標準化される |
| システム分類 | 記録ソフト | 支援知識生成システム |
| 将来性 | 記録の蓄積のみ | AI分析への接続 |
