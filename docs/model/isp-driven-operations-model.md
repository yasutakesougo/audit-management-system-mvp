## Abstract

本文書は、障害福祉サービスの支援業務を構造化するための運用モデル  
**「ISP-Driven Welfare Operations Model」** を定義する。

このモデルは、個別支援計画（ISP: Individualized Support Plan）を中心に据え、  
日次の記録から月次の計画見直しまでを **1 本の PDCA パイプライン** として統合する。

従来の福祉 IT システムが「記録の電子化」に留まるのに対し、  
本モデルは **記録 → 分析 → 提案 → 人の判断 → 知識蓄積** の全循環を設計対象とする。

核心的な主張は以下の 3 点である。

1. **ISP を起点とした三層構造** が、支援の Why / How / Do を分離する
2. **Observation-Insight-Proposal パイプライン** が、記録を判断材料に変換する
3. **Human-in-the-Loop 設計** が、AI 提案と人の裁量のバランスを取る

## 1. Introduction — 福祉 DX の課題

### 現状の問題

障害福祉サービスの現場では、以下のような課題が存在する。

| 課題 | 状況 |
|------|------|
| **記録の断絶** | 日々の記録が蓄積されるだけで、計画の見直しにつながらない |
| **暗黙知の属人化** | ベテランの判断基準が共有されず、担当交代で失われる |
| **計画と実施の乖離** | ISP が作成時点で固定され、現場の変化が反映されない |
| **分析の不在** | 記録データはあるが、パターンや傾向が可視化されない |
| **入力負担の増大** | 分析精度を求めて入力項目を増やすと、現場が離脱する |

### 既存システムの限界

```
従来の福祉 IT システム:

  記録 → 保存 → 検索 → (終わり)

  問題:
  ・記録が保存で止まる
  ・分析は別ツール or 手作業
  ・計画への反映は担当者の記憶に依存
  ・判断の根拠が残らない
```

### 本モデルが目指すもの

```
ISP-Driven Operations Model:

  記録 → 整理 → 分析 → 提案 → 人の判断 → 反映 → 知識蓄積
                                                    ↓
                                              次のサイクルへ
```

## 2. ISP-Driven Operations Model とは

### 定義

> **ISP-Driven Operations Model** は、  
> 個別支援計画（ISP）を中心に据え、支援業務の全過程を  
> 構造化された PDCA サイクルとして運用するモデルである。

## 3. 三層構造モデル

### ISP の三層分離

支援における「なぜ（Why）」「どうやって（How）」「実行（Do）」を明確に分離する。

```
          抽象度 高
            ↑
  ┌─────────────────────┐
  │   L1: ISP            │  ← 生活全体の方向性
  │   (Why)              │     長期目標・ニーズ・意向
  │                      │     更新: 低頻度（年1〜2回）
  └──────────┬──────────┘
             │ Bridge 1: Assessment → L2
             ▼
  ┌─────────────────────┐
  │   L2: 支援計画シート  │  ← 行動レベルの設計
  │   (How)              │     支援方針・具体的対応・環境調整
  │                      │     更新: 中頻度（90日サイクル）
  └──────────┬──────────┘
             │ Bridge 2: L2 → L3
             ▼
  ┌─────────────────────┐
  │   L3: 手順書兼記録   │  ← 手順化 + 実施記録
  │   (Do + Record)      │     ステップ・チェック・記録
  │                      │     更新: 高頻度（毎日）
  └─────────────────────┘
          抽象度 低
            ↓
```

### なぜ三層に分けるのか

| 問題 | 三層で解決 |
|------|----------|
| ISP が変わると全手順を書き直す必要がある | 各層が独立しているため、変更は該当層のみ |
| 手順書に「なぜ」が書かれていない | L1 → L2 → L3 のトレースで根拠がたどれる |
| 細かい手順変更が ISP に影響する | L3 の変更は L1 に伝播しない |
| 新人が「なぜこの手順か」を理解できない | Bridge の Provenance で出典を表示 |

### ブリッジの役割

ブリッジは層間のデータ変換を行う **純粋関数** である。

```
Bridge 1: Assessment → L2
  入力: ICF 分類 + 特性アンケート
  出力: 支援計画シートのフィールド候補
  特徴: 候補提示のみ。人が確認して採用する

Bridge 2: L2 → L3
  入力: 支援方針 + 具体的対応 + 環境調整
  出力: 手順ステップの候補
  特徴: 重複排除 + Provenance 付き

Bridge 3: Monitor → L2
  入力: モニタリング評価結果
  出力: 支援計画の更新候補
  特徴: 自動追記（確認のみ） + 候補提示（チェックボックス選択）
```

## 4. Observation-Insight-Proposal パイプライン

### 概要

現場の記録を「判断材料」に変換する 3 段階のパイプライン。

```
Observation              Insight                   Proposal
(何が起きたか)            (何を意味するか)            (何をすべきか)
┌──────────┐    ┌──────────────────┐    ┌──────────────────────┐
│ Daily    │    │ パターン検出     │    │ 「午後帯の離席が     │
│ ABC      │──▶│ 傾向分析        │──▶│  増加しています。     │
│ Handoff  │    │ 変化検出        │    │  休憩導線の見直しを   │
│ Monitor  │    │ 進捗評価        │    │  候補として確認して   │
│          │    │                  │    │  ください」           │
└──────────┘    └──────────────────┘    └──────────────────────┘

  入力最小化            純粋関数            候補提示 + 根拠
  (原則 2)            (テスト容易)          (原則 3 + 4)
```

### Observation → Insight の変換例

| Observation（観察） | Insight（洞察） |
|---------------------|----------------|
| ABC 記録 15 件（午後帯） | 午後帯に離席行動が集中 (+35%) |
| 申し送り「落ち着かない」×5 回 | 繰り返しパターン検出（環境要因の可能性） |
| 目標 A の達成率 30% | 目標達成が停滞（前月同水準） |
| モニタリング「一部有効」 | 支援方針の部分的見直しが必要 |

### Insight → Proposal の変換ルール

提案は常に以下の形式を取る。

```
提案本体:   何を提案しているか
根拠:       なぜその提案が出たか
  - 対象期間:   2026-02-15 〜 2026-03-01
  - 参照記録数: 23件
  - 前月比:     +35%
  - 使用ルール: ABC 頻度ルール v2
参照元:     どのレコードに基づくか
  - abc-0042, abc-0038, abc-0035, ...
```

## 5. 福祉 PDCA エンジン

### 従来の PDCA と本モデルの PDCA

```
従来の PDCA:
  Plan → Do → Check → Act
  (紙で計画 → 実施 → 会議で評価 → 計画修正)
  問題: Check と Act が形骸化しやすい

ISP-Driven PDCA:
  Plan → Do → Check → Act
  (三層計画 → 記録付き実施 → 自動分析+提案 → 人が判断して反映)
  解決: Check を自動化し、Act を候補提示で支援
```

### PDCA サイクルの詳細

```
PLAN ─────────────────────────────────────────────
  L1: ISP で方向性を定める
  L2: 支援計画シートで行動設計する
  L3: 手順書に手順化する
  ↓
DO ───────────────────────────────────────────────
  Daily: 手順に沿って支援を実施し、記録する
  ABC: 特定の行動について機能分析する
  Handoff: シフト引き継ぎで申し送る
  ↓
CHECK ────────────────────────────────────────────
  Normalization: 記録データを構造化する
  Insight Engine: パターン・傾向・変化を検出する
  Monitoring: 90 日サイクルで支援を評価する
  ↓
ACT ──────────────────────────────────────────────
  Proposal Engine: 見直し候補を生成する
  Human Decision: 人が採用/却下を判断する
  Execution: 判断を L2 支援計画に反映する
  Knowledge: 判断履歴を組織知として蓄積する
  → PLAN へ戻る
```

### PDCA の時間軸

| サイクル | 頻度 | 内容 |
|---------|------|------|
| **日次ループ** | 毎日 | 記録入力 → 朝会サマリー → 今日の注意点 |
| **週次ループ** | 毎週 | 申し送り傾向 → 繰り返しパターン検出 |
| **月次ループ** | 90 日 | モニタリング → 目標進捗評価 → 計画見直し候補 |
| **年次ループ** | 年 1〜2 | ISP 見直し → アセスメント再実施 → 全体方針更新 |

## 6. Human-in-the-Loop 判断モデル

### なぜ AI に自律判断させないのか

福祉業務において AI の自律決定が不適切である理由。

| 理由 | 説明 |
|------|------|
| **制度的制約** | 個別支援計画の変更は支援者・管理者の判断で行うことが制度上求められる |
| **倫理的制約** | 利用者の生活に直接影響する判断を機械に委ねることは倫理的に問題がある |
| **文脈依存** | 支援判断は記録に現れない文脈（家族状況、体調の微妙な変化）に依存する |
| **信頼構築** | 現場が「使わされている」と感じると、入力品質が低下し、システム全体が機能しなくなる |

### 判断モデルの構造

```
      System の責務                 Human の責務
  ┌─────────────────┐        ┌─────────────────────┐
  │ データ収集       │        │                     │
  │ 構造化          │        │                     │
  │ パターン検出    │        │                     │
  │ 候補提示       │───────▶│ 確認・検討           │
  │                │        │ 文脈の補完           │
  │                │        │ 採用/却下の判断      │
  │ 判断の記録     │◀───────│ 判断の実行           │
  │ 履歴の蓄積     │        │                     │
  │ パターン学習   │        │                     │
  └─────────────────┘        └─────────────────────┘
```

### 提案の位置づけ

| ❌ 提案ではないもの | ✅ 提案であるもの |
|-------------------|-----------------|
| 命令 | **見直し候補** |
| 正解 | **下書き候補** |
| 結論 | **注意候補** |
| 指示 | **比較材料** |

## 7. 組織知化モデル

### 暗黙知から形式知へ

```
現在の福祉現場:
  ベテラン A の頭の中  → 「なんとなくこうする」
  新人 B              → 「何を基準に判断すればいいか分からない」
  担当交代            → 「知識が消える」

ISP-Driven Model 適用後:
  判断履歴            → 「この状況では、過去にこう判断した」
  採用パターン        → 「この提案は 80% 採用されている」
  却下理由            → 「この提案が却下される理由は主に X」
  結果追跡            → 「採用後、Y が改善した」
```

### 組織知化の 4 段階

```
Stage 1: 判断の記録
  ・採用/却下を ProposalAdoptionRecord として保存する
  ・却下理由を構造化して収集する
     ↓
Stage 2: パターンの発見
  ・ソース別の採用率を分析する
  ・却下理由の傾向を可視化する
  ・有効だった提案のパターンを抽出する
     ↓
Stage 3: 知識の構造化
  ・判断事例をライブラリ化する
  ・新人教育の材料として利用する
  ・引き継ぎドキュメントを自動生成する
     ↓
Stage 4: 組織の進化
  ・提案品質を改善する（採用パターンに基づく）
  ・法人内の他事業所に横展開する
  ・業界標準のベストプラクティスとして共有する
```

### 知識蓄積の構造

```
                  ┌─────────────────────────────┐
                  │     Knowledge Repository     │

                  │  ┌──────────────────────┐   │

                  │  │ Records              │   │
                  │  │ ・採用/却下履歴       │   │
                  │  │ ・却下理由           │   │
                  │  │ ・反映フィールド     │   │
                  │  └──────────┬───────────┘   │
                  │             │               │
                  │  ┌──────────▼───────────┐   │
                  │  │ Provenance Trail     │   │
                  │  │ ・出典追跡           │   │
                  │  │ ・変更履歴           │   │
                  │  │ ・版管理             │   │
                  │  └──────────┬───────────┘   │
                  │             │               │
                  │  ┌──────────▼───────────┐   │
                  │  │ Organizational       │   │
```

## 8. 設計原則との関係

ISP-Driven Operations Model の各要素が、設計原則 10 箇条のどれに対応するか。

| モデル要素 | 対応原則 | 関係 |
|-----------|---------|------|

| **Bridge** | 原則 4（根拠付与） | Provenance で層間の出典を追跡する |

| **Operational UI** | 原則 1（現場導線主体） | 表はシンプル、裏は強い |
| **Human Decision** | 原則 5（人間決定） | 最終判断は必ず人が行う |
| **Knowledge** | 原則 8（組織知化） | 履歴を組織の判断資産にする |
| **日次ループ** | 原則 7（日次優先） | まず毎日の価値から定着させる |
| **一貫した設計** | 原則 9（一貫性） | 全画面が同じ思想で動く |

## 10. 評価フレームワーク

### 本モデルの成功基準

モデルの有効性は以下の 5 軸で評価する。

| 評価軸 | 測定方法 | 目標 |
|--------|---------|------|
| **判断速度** | 見直し判断にかかる時間 | 従来の 50% 以下 |
| **自然な振り返り** | 月次見直しの実施率 | 90% 以上 |
| **支援品質の均質化** | 職員間の支援方針のばらつき | 標準偏差 30% 減 |
| **説明可能性** | 判断根拠を提示できる割合 | 100% |
| **知識の蓄積** | 組織知化された判断事例数 | 月 20 件以上 |

### 段階的評価

| 段階 | 指標 | 達成条件 |
|------|------|---------|
| **Level 1: 定着** | 現場の日次利用率 | 80% 以上が毎日使用 |
| **Level 2: 循環** | PDCA サイクルの完走率 | 90 日サイクルの見落としゼロ |
| **Level 3: 知化** | 採用パターンの蓄積 | 判断事例 100 件以上 |
| **Level 4: 展開** | 他事業所への横展開 | 2 事業所以上で導入 |

## 11. 既存アプローチとの比較

| 観点 | 従来の福祉 IT | EHR (電子カルテ) | 本モデル |
|------|-------------|----------------|---------|
| **中心概念** | 記録の保存 | 診療記録の管理 | ISP 起点の PDCA |
| **分析** | なし or 手作業 | 統計レポート | 自動パイプライン |
| **提案** | なし | 薬剤アラート | 構造化候補 + 根拠 |
| **判断支援** | なし | 診断支援 (CDS) | Human-in-the-Loop |
| **知識蓄積** | なし | 症例データベース | 判断パターンの組織知化 |
| **入力負担** | 紙→電子の置換 | 構造化入力 | 最小化 + 自動補完 |
| **計画との接続** | 独立 | 独立 | Bridge で自動連 |
| **出典追跡** | なし | 監査ログ | Provenance 全フィールド |

### 本モデルの独自性

1. **三層分離** — Why / How / Do を明確に分離し、Bridge で接続する
2. **提案は候補** — AI は提案するが決定しない。根拠を必ず添える
3. **日次価値優先** — 月次の大きな価値より、毎日の小さな価値を先に出す
4. **組織知化** — 個人の判断を組織の資産に変換する構造を内蔵する

## 12. 応用可能性

本モデルは障害福祉に限定されず、以下の領域にも応用可能と考えられる。

| 領域 | 応用ポイント |
|------|------------|
| **高齢者介護** | ケアプランの三層化 + 日次記録からの見直し候補 |
| **特別支援教育** | 個別の教育支援計画 (IEP) の PDCA 化 |
| **リハビリテーション** | リハビリ計画 → 実施記録 → 評価 → 計画見直し |
| **精神科医療** | 治療計画の段階的見直し + 行動パターン分析 |
| **ソーシャルワーク** | ケースワークの構造化 + 組織知化 |

### 共通要素

これらの領域に共通する要素は以下である。

- 個別の計画（ISP / ケアプラン / IEP）が存在する
- 日次の記録が蓄積される
- 定期的な見直しが制度上求められる
- 判断は専門家（人）が行う
- 組織としての知識蓄積が課題である
