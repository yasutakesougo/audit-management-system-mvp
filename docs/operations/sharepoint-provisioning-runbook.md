# Audit Management System MVP SharePoint Provisioning / Drift Recovery Runbook

調査実施日: 2026年6月19日
対象コミット: `7dde895340c5a6914f2c12ecd858621d5821a105`

## 1. 目的
本手順書は、SharePoint リストの構成不整合（列の欠落や名称不一致などのスキーマ・ドリフト）、接続品質の劣化、および書き込み停止が発生した際、システム管理者および運用保守担当者が迅速に調査、復旧、または開発者へのエスカレーションを行うための手順を定義するものです。

## 2. 対象読者
* システム管理者
* 運用保守担当オペレーター
* 拠点内の IT サポート担当者

## 3. SharePoint 側で起きる主な異常
SharePoint 連携において発生する主なトラブルの分類と原因は以下の通りです。
* **スキーマの乖離（スキーマ・ドリフト）**: リストの新規作成時や改修時に、必須の列（フィールド）が物理的に不足している状態、あるいは列の内部名（物理名）に数字サフィックス（例: `FullName0` 等）が自動付与されて不一致が発生している状態。
* **接続・通信の障害**: 認証設定（MSAL）の期限切れ、接続先サイト URL の指定誤り、一時的な API 制限（Throttling）やネットワーク瞬断による通信不可。
* **書き込み保護（無効化）**: 保守期間中やテスト環境でのデータ保護のため、意図的にアプリ側のデータ保存処理がブロックされている状態。

---

## 4. 画面上の警告と意味
アプリが検知した SharePoint の状況は、画面上にバナーやバッジとして可視化されます。

### 1. 書き込みが無効です（オレンジ色の警告バナー）
* **表示契機**: 環境変数定義で書き込み制限フラグが `0`（無効）に設定されている場合、全画面の最上部に固定表示されます。
* **影響範囲**: データの保存・更新・削除などの操作がすべてアプリによってブロックされます（読み取り専用での動作継続）。

### 2. データの同期に時間がかかっています（赤色のエラーバナー）
* **表示契機**: 設定ファイルの接続先情報不足や、認証失敗、および必須リストに到達できない重大な接続エラー（`action_required` または `critical`）が発生した場合に表示されます。
* **影響範囲**: データが表示されなくなるか、古いキャッシュデータのみで動作する「同期機能劣化」状態になります。

### 3. ヘッダー部のステータスバッジ（管理者専用）
ヘッダー右上の管理者向けエリアに、問題の深刻度に応じたバッジが表示されます。クリックするとポップオーバーで推奨対処ガイドが表示されます。
* **SP Watch（黄色）**: 一部の列で名前のドリフト（自動解決済み）や、重要度の低いオプション列の不足を検出した状態。アプリの操作は継続可能です。
* **SP Action（オレンジ色）**: ガバナンス用の子リスト初期化失敗など、対応が推奨される状態。
* **SP Critical（赤色）**: 必須列の欠落や、接続不可により一部の業務機能が停止している状態。

## 5. 初動確認フロー
管理者あてにエラー報告が入った場合、以下の手順で状況を一次切り分けします。

```
【管理者初動フロー】
  ① 管理者画面（/admin/status）を開く
       ↓
  ② 異常カード（FAIL 表示など）の有無を確認
       ↓
  ③ ブラウザ開発者ツール（Console / Network タブ）を確認
       ↓
  ④ トラブル別の対応へ（Drift/接続劣化/認証エラー等）
```

1. **管理者ステータス画面のチェック**:
   管理者権限でアプリにサインインし、`/admin/status` にアクセスします。PASS、WARN、FAIL の判定カードから、どのリスト・どのカテゴリ（config, auth, connectivity, lists, schema, permissions）で問題が起きているかを確認します。
2. **ブラウザコンソールのログチェック**:
   F12キーでブラウザ開発者ツールを開き、`Console` タブにエラー（特に SharePoint 関連の API 失敗）が出力されていないか確認します。

## 6. Drift（構成の乖離）検知時の対応
アプリの起動時に自動でスキーマ（列構成）の検査が実行されます。

### 1. スキーマ自動吸収の仕組み
* 列の内部名（物理名）にサフィックスが付与されている場合（例: `cr013_requiresToiletGuidance0`）、アプリは事前に定義された「候補名リスト」から自動で物理名を見つけ出してマッピングを解決します。
* これにより、画面上で `SP Watch（黄色）` 警告が出ていても、アプリは正常に読み書きできます。この場合は、緊急の復旧作業は不要です。

### 2. 手動対応が必要なケース（FAIL (essential) 状態）
* システムの稼働に必須の列（例: 記録者IDや日付など）が物理的にリストに存在せず、候補名解決もできなかった場合は、診断画面で `FAIL` カードが表示され、データの保存や取得が停止します。
* **対応**: 後述の「Provision の再実行」を行ってスキーマを修復します。

## 7. Provision（初期化・修復）再実行の判断基準
Provision は、SharePoint に必要なリストや列を自動で再作成・修復する機能です。再実行するかどうかの判断は以下に従います。

### 再実行すべきケース (自動修復可能)
* 新規導入時、または SharePoint サイト側のリスト自体が物理的に存在しない場合。
* 必須列が不足していると警告され（FAIL 表示）、リスト自体には列制限の上限に余裕がある場合。
* アプリのアップデートにより、システムが新しい必須列を追加で必要としている場合。

### 再実行を控えるべきケース
* **SharePoint の列数・容量制限超過時**: SharePoint リストには「8KB 行サイズ制限」および「インデックス数上限（最大20件）」があります。上限に達している状態で Provision を実行すると、列の作成が失敗しエラーログが出続けます。この場合は再実行の前に、SharePoint サイト側で不要な列や古いインデックスを手動でパージ（削除）する必要があります。
* **認証（MSAL）そのものが失敗している場合**: アプリが SharePoint にログインできていないため、Provision を実行しても接続エラーになります。まずは認証設定を確認してください。

## 8. Write Disabled / Connection Degraded 時の対応

### 1. 書き込み制限（Write Disabled）時の対応
* **状況**: アプリ上の保存ボタン等を押してもエラーになる、または警告バナーが表示されている。
* **初動**: 管理者画面で設定を確認し、環境設定の「書き込み許可設定」がオフ（`0`）になっていないかチェックします。テスト・開発環境等で安全のために意図的にオフにしている場合を除き、本番環境ではオンにする必要があります。

### 2. 接続劣化（Connection Degraded）時の対応
* **状況**: 画面最上部に「データの同期に時間がかかっています」バナーが表示され、一覧が空になる。
* **初動**: 
  1. `管理者ステータス（/admin/status）` 画面で、`auth` や `connectivity` が PASS になっているか確認します。
  2. `auth` が FAIL の場合は、ブラウザで一度サインアウトし、管理者アカウントで再度サインイン（MSAL 認証）を試します。
  3. テナント全体の通信障害の可能性もあるため、Microsoft 365 サービス正常性ダッシュボードで SharePoint Online の障害情報を確認します。

## 9. 復旧管理テーブル（管理者向けクイックリファレンス）

| 状況 | 画面表示 | 原因候補 | 初動対応 | エスカレーション |
| --- | --- | --- | --- | --- |
| **Drift warning** | Drift/Schema 警告 | 列不足・内部名不一致 | JSON確認、対象リスト確認 | 管理者/開発者 |
| **Write Disabled** | 書き込み停止表示 | 安全保護・接続不良 | 入力停止、管理者確認 | 管理者 |
| **Connection Degraded** | 接続劣化表示 | SP/API/認証不安定 | 再読込、時間を置く | 継続時に開発者 |
| **Provision failure** | 初期化失敗 | 権限不足・列作成失敗 | 権限確認、ログ共有 | 管理者 |
| **0件取得** | 一覧が空 | フィルタ/日付/接続/権限 | 日付・対象者・接続確認 | 再現時に調査 |

## 10. 開発者へ渡す情報
管理者の調査で復旧しない場合は、以下の情報を収集して速やかにシステム開発者へ連携してください。
1. **診断 JSON データ**:
   `/admin/status` にある「診断詳細のダウンロード」または「診断結果をコピー」ボタンから、判定結果の JSON データを取得し共有します。
2. **アプリの環境設定パラメータ**:
   アプリの起動時の設定（接続先サイトURL、認証設定など）のテキストコピー。
3. **ブラウザのエラーコンソールログ**:
   F12キーの `Console` タブのログ全体（赤文字のエラー行をすべて含む形）のスクリーンショット、またはログの書き出しファイル。
4. **対象リストの構造スクリーンショット**:
   エラーが出ている SharePoint リストの「リストの設定」画面から、列の一覧およびインデックス付きの列の構成がわかるスクリーンショット。

## 11. 現場職員への案内文（告知テンプレート）
障害発生時および復旧完了時に、現場のチャットツール等で職員に案内する文面のテンプレートです。

### 1. 障害発生時のアナウンス
> 💡 **【お知らせ】SharePointデータ連携の一時遅延について**
> 
> 現在、記録システムのデータ接続において一時的な遅延またはエラーが発生しています。
> * **対象**: 全記録機能（キオスクを含む）
> * **状態**: 画面上部に「データの同期に時間がかかっています」または「書き込みが無効です」と表示される場合があります。
> * **職員の皆様へのお願い**:
>   - データの保存ボタンを押した際にエラーが出た場合、保存されている可能性がありますので、何度も連打せず、一度「履歴一覧」に記録が表示されているかご確認ください。
>   - 画面が動かない場合は、ブラウザの再読み込みをお試しください。復旧までしばらくお待ちいただけますようお願いいたします。

### 2. 復旧完了時のアナウンス
> 💡 **【復旧】SharePointデータ連携の正常化について**
> 
> 遅延が発生していたデータ連携が復旧し、現在は正常に保存・読み込みが可能です。
> * **職員の皆様へのお願い**:
>   - 接続エラーのバナーが消えていることをご確認ください。
>   - 万が一、まだエラーバナーが表示されている場合は、お手数ですが一度ブラウザの「再読み込み」を行った上でご利用ください。
>   ご協力ありがとうございました。

## 12. 復旧後の確認チェックリスト
復旧作業が完了した際、管理者が以下のチェックを行い、システムの完全正常化を判定します。

- [ ] **管理者ステータス判定**: `/admin/status` ですべてのチェック項目が PASS（または許容された WARN）になり、FAIL カードが 0 件になっているか。
- [ ] **バナー非表示確認**: ホーム画面や記録入力画面から、警告バナーや同期エラーメッセージが消えているか。
- [ ] **テストデータの書き込み**: テスト用利用者において、簡単な日常記録がエラーなく保存され、一覧に追加されるか。
- [ ] **テストデータの読み込み**: 保存したテストデータおよび過去の記録が、一覧や履歴欄に正しく表示されるか。
- [ ] **終了操作の確認**: キオスクモード等の各画面から、長押し終了メニューなどの導線が問題なく機能するか。

1. **`admin: add-diagnostic-json-download` (小規模)**
   * **目的**: 管理者画面から診断結果（`StabilitySummary` などの詳細情報）をワンクリックで JSON ファイルとしてローカルに保存できるようにし、開発チームへのエスカレーションスピードを向上させる。
2. **`health: add-automatic-remediation-dryrun` (中規模)**
   * **目的**: 不足している列がある場合、実際に物理構成を書き換える前に、どのような変更が加えられるかをプレビュー表示する「ドライラン（シミュレーション）機能」を管理者画面に追加する。
3. **`observability: add-drift-metric-exporter` (小規模)**
   * **目的**: スキーマ・ドリフトや一時的な接続劣化のシグナル発生回数をアプリケーションログに明示的に出力し、DatadogやApplication Insights等の監視システムと自動連携できるようにする。
