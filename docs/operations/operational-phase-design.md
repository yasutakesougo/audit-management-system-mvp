# 🧭 運用フェーズ設計：運用権限・Nightlyフロー手順書

本文書は、Audit Management System MVP の本番運用開始にあたり、「誰が何をするのか」「異常時に誰がどう動くのか」を定義した実行ガイドです。

---

## 🔐 1. 運用権限マトリクス (Role-Based Access Control)

システムの「強力な力」を安全に制御するため、ユーザーを以下の4つのロールに分類し、ADグループ（Group ID）に基づき制御します。

### 1-1. ロール定義

| ロール | 想定対象者 | 判定基準 (AD Group) |
| :--- | :--- | :--- |
| **A. 現場利用者** | 一般職員 | `viewer` (デフォルト) |
| **B. 現場リーダー** | 現場主任・管理者 | `reception` (`VITE_RECEPTION_GROUP_ID`) |
| **C. 運用管理者** | 施設長・IT担当 | `admin` (`VITE_SCHEDULE_ADMINS_GROUP_ID`) |
| **D. システム保守者** | 開発エンジニア | `admin` + 特権操作権限 |

### 1-2. 機能アクセス制限

| 機能 | 現場利用者 | 現場リーダー | 運用管理者 | システム保守者 |
| :--- | :---: | :---: | :---: | :---: |
| **日々の記録入力・閲覧** | ○ | ○ | ○ | ○ |
| **自分の記録の修正** | ○ | ○ | ○ | ○ |
| **他者の記録の修正/承認** | × | ○ | ○ | ○ |
| **マスタ（利用者等）の閲覧** | ○ | ○ | ○ | ○ |
| **マスタの編集・更新** | × | × | △ (2名確認) | ○ |
| **管理・診断ダッシュボード** | × | × | ○ | ○ |
| **Nightly Patrol 結果確認** | × | × | ○ | ○ |
| **SharePoint 構造変更** | × | × | × | ○ |
| **データのアーカイブ・一括削除** | × | × | × | ○ (要承認) |

### 1-3. 例外・代行ルール (緊急時の対応方針)

*   **運用管理者不在時の対応**: 管理者不在時、破壊的操作（削除・マスタ更新）は原則**実施禁止**とします。
*   **技術代行**: 緊急かつ不可避な修正が必要な場合のみ「システム保守者」が操作を代行し、完了後に Teams チャネルへ事後報告を行います。
*   **承認の証跡**: △操作（マスタ更新など）は、Teams 等の公式チャネルでの「依頼・承認」のテキスト記録をもって、2名確認の承認とみなします。

## 🚦 2. Nightly Patrol：人間系運用フロー

「ログは完璧でも、誰も見ていない」状況を防ぐための、毎朝のルーチン定義です。

### 2-1. アラート分類と対応レベル

Nightly Patrol (`npm run patrol`) の結果を以下のように分類します。

| レベル | 判定基準例 (Patrol結果) | 対応アクション | 責任者 |
| :--- | :--- | :--- | :--- |
| **🟢 Level 1: Watch** | 巨大ファイルの微増、TODOの追加、一時的な retry | 当日中の確認とログ記録。連続発生時は格上げ検討。 | 監視当番 |
| **🟡 Level 2: Action** | 通信エラー(Failed)の発生、特定画面のデータ欠損(Drift) | 午前中に原因特定。回避策を現場リーダーへ通知。 | 運用管理者 |
| **🔴 Level 3: Critical** | 記録不可、スキーマ不整合、認証エラー(Login Fail) | **即時エスカレーション**。現場に「代替運用（紙）」を発動。 | システム保守者 |

### 2-2. アラートの「無視・格上げ」基準

監視当番の疲弊を防ぎ、重要な予兆を見逃さないための判断基準です。

*   **無視（Watch継続）してよい条件**:
    *   単発（当日のみ）の発生である。
    *   前日まで正常であり、かつ業務画面の動作に影響が出ていない。
    *   既にシステム保守者が「既知の問題」として共有済みである。
*   **格上げ（Level 2へ）の条件**:
    *   **3日連続**で同じ Watch 項目が発生している（潜在的な累積リスク）。
    *   エラー件数が昨日の2倍以上に急増している。

### 2-2. 朝の 9:15 ルーチン

1.  **08:30 - 確認**: 監視当番が `docs/nightly-patrol/` の最新レポート (`.md` / `.json`) を確認。
2.  **09:00 - 分類**: 異常がある場合、上記表に基づき Level 1〜3 に分類。
3.  **09:15 - 通知**: Teams チャネルに以下のフォーマットで投稿。

> **【Nightly Patrol 朝報】**
> *   **状態**: 🟡 Action Required (データ連携遅延)
> *   **影響**: 設定マスタの一部が反映されていません。
> *   **対応**: 10時までに解消予定。それまでの入力は「保存」せず「下書き」を推奨。
> *   **担当**: 〇〇

## 🚑 3. 障害時レスキュー・アクションカセット

システムが「赤 (Critical)」判定になった際の、現場向けの緊急初動ガイドです。

### 3-0. 【厳禁】善意の破壊を防ぐための禁止事項

障害発生時、現場および管理者は以下の操作を**絶対に行わない**でください。

*   **SharePoint 側の直接編集**: リストの列名変更、行の手動削除。
*   **設定管理画面の不用意な操作**: Feature Flag のオンオフ、リストタイトルの変更。
*   **自己判断での再起動・再デプロイ**: 状態をさらに悪化させ、原因究明を困難にするリスクがあります。
*   **データの削除・上書き**: 「とりあえず消してやり直す」は、監査証跡を破壊します。

> [!CAUTION]
> すべての復旧作業は、**システム保守者の指示**に従って実施してください。

### 3-1. 現場リーダーの 3 ステップ

1.  **止める**: システムへの入力を中断するよう全職員に指示。
2.  **切り替える**: 準備済みの「代替運用セット（紙の支援日誌）」を配布。
3.  **繋ぐ**: システム保守者へ「いつ、何が起きたか」を報告。

### 3-2. 復旧後のデータ同期ルール
*   **短時間（1時間以内）**: 職員が手動で転記（後から入力）。
*   **長時間（1日以上）**: 入力負担を考慮し、重要な監査項目のみ入力する「簡易同期モード」を運用管理者が許可する。

## 🛠 4. 導入 3 週間プランの成功指標 (behavior-centric KPIs)

| 週 | フェーズ | 成功の定義 (システムの安定ではなく「人の行動」) |
| :--- | :--- | :--- |
| **第1週** | **パイロット** | 2名の職員のうち1名以上が、**自発的に**システムを使って記録を完了させる。 |
| **第2週** | **Pruning (削ぎ落とし)** | 現場で「一度も触られなかったボタン」や「混乱を招いた表示」を具体的に特定し、削減リストを作成する。 |
| **第3週** | **自律運用** | **「一度も紙の予備運用に戻ることなく」** 1日すべての業務をシステムで完結できる日が1日以上発生する。 |
