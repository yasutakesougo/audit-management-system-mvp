# ナビゲーションOS設計書 (Navigation IA)

本システムのサイドメニューは、単なるリンク集ではなく、ユーザーの役割や業務状況（キオスク、フォーカスなど）に応じて最適な業務面を提示する「ナビゲーションOS」として設計されています。

## 1. 情報設計 (7-Group IA)

ナビゲーション項目は必ず以下の7つのグループのいずれかに属します。順番は業務フローに沿っています。

| グループキー | 表示名 | 設計意図・対象業務 |
| :--- | :--- | :--- |
| `today` | Today | **現場の当日実行**: スケジュール確認、送迎、日中の記録、申し送り等 |
| `records` | Records | **記録・振り返り**: 過去の記録の閲覧、サマリー確認、月次業務日誌等 |
| `planning` | Planning | **支援計画・評価**: アセスメント、ISP（個別支援計画）の作成・更新、分析等 |
| `operations` | Operations | **拠点運営**: 請求、勤怠管理、コンプライアンス報告、リソース管理等 |
| `billing` | Billing | **請求**: 請求処理に特化したワークスペース |
| `master` | Master | **マスタ管理**: 利用者マスタ、職員マスタ等 |
| `platform` | Platform | **管理基盤**: システム設定、自己点検、ログ確認等の管理機能 |

## 2. 表示制御のエントリポイント (Navigation Contract)

ナビゲーションの表示判定は、UIコンポーネント内で行わず、必ず `src/app/config/navigationConfig.helpers.ts` の **`buildVisibleNavItems`** を通すこと。これが Navigation OS の唯一の実行契約（Contract）である。

## 3. 表示制御の5層フィルター

サイドメニューの項目は、以下の順序でフィルタリング（`buildVisibleNavItems` 内で実行）され、最終的な露出が決定されます。

1. **Role フィルター (`audience`)**: 利用者の権限（Admin, Staff, Reception）に基づいて項目を絞り込む。
2. **Feature Tier (`tier`)**: `Core` (常時), `More` (トグル内), `Admin` (管理者エリア) に分類。
3. **Kiosk モード**: 現場端末（キオスク）では `today` グループ以外を非表示にし、誤操作を防止する。
4. **Focus モード**: 全画面表示が必要な場合、サイドメニュー全体を一時的に隠蔽する。
5. **User Preference**: ユーザーが設定画面から特定のグループや項目を個別に非表示にできる。

## 4. 変更・追加ルール

新規画面を追加する際は、以下のチェックリストを遵守してください。

- [ ] **グループの選定**: 上記7グループのどこに属すべきか？（「とりあえず末尾」は禁止）
- [ ] **権限の設定**: 誰が見るべきか？ (`all`, `staff`, `admin`, `reception`)
- [ ] **ティアの選定**: 頻繁に使う `core` か、たまに使う `more` か？
- [ ] **検索性の確保**: `label` が直感的か？（メニュー検索でヒットするか）
- [ ] **回帰テストの更新**: `tests/unit/app/config/navigationConfig.spec.ts` にテストを追加したか。

## 5. キオスクモード・フッターメニュー構造整理

キオスクモードにおけるフッターメニューおよびナビゲーション周辺は、タブレット端末での操作性向上のため、以下の3つの主要コンポーネントが疎結合に連携する構成となっています。

*   **`AppShell.tsx` での表示制御**: `isKioskMode` 判定により、標準のヘッダー・サイドバーを非表示にし、キオスク専用ナビゲーションと離脱用FABを描画します。また、メニューによる被りを防ぐため、下部余白（`contentPaddingBottom={isKioskMode ? 160 : undefined}`）を設定しています。
*   **`KioskNavigation.tsx` (フッターメニュー本体)**: ホーム、予定、通所、記録、支援手順、トイレ確認への画面遷移を担当する下部固定の共通メニューです。申し送りや受電ログなどのダイアログ表示機能は、直接依存せずグローバルカスタムイベント（`window.dispatchEvent`）を介して起動します。
*   **`FooterQuickActions.tsx` (ダイアログホスト)**: キオスクモード時（`onlyDialogs={true}`）は、ナビゲーションバー自体を非表示にし、「今すぐ申し送り」ダイアログや「受電ログ」ドロワーの実体と表示状態の管理に専念します。
*   **`KioskExitFab.tsx` (離脱防止)**: 通常モードへの意図しない切り替えを防止するため、1.5秒（1500ms）の長押し（Long-press）を要求するフローティングボタンです。長押し成功時にのみアクションメニューが表示されます。
*   **`appendKioskSearchParams()` によるパラメータ自動クレンジング**: キオスクモード内（`/kiosk`）での遷移時、前ユーザーの選択状態や操作ステップが残留して表示・記録の不整合（状態の交錯）が発生するのを防ぐため、`userId`, `user`, `slotId`, `step` を自動クレンジング（除去）します。一方で、`kiosk=1` や `provider=sp` などの動作モード用パラメータは安全に維持・伝播されます。

## 6. キオスクモードにおける誤操作・情報混同防止対策（安全対策）

共有タブレット端末として複数職員・利用者が交代で操作することを前提とし、以下の3つの安全設計を徹底しています。

1.  **通常モードへの意図しない切り替え防止**:
    現場用のキオスク画面から通常モードへ意図せず切り替わることを防ぐボタン（`KioskExitFab`）には、長押しアクションを要求するガードが設けられています。これにより、利用者の誤タップや不要な画面離脱を物理的に防止します。
2.  **URLパラメータの自動クレンジングによる情報混同防止**:
    端末が共有された際、クエリパラメータに古いユーザーIDやコンテキスト（手順スロット、進捗ステップなど）が残留すると、他の利用者の画面への遷移がロックされたり、他者の記録を上書きしてしまうリスクがあります。`appendKioskSearchParams()` を用いて遷移時にこれらの特定パラメータのみを自動クレンジングすることで、情報の競合・リークを防ぎ、堅牢なデータ整合性を維持します。
3.  **一時入力データの保護（画面状態の維持）**:
    「申し送りクイック入力」や「受電ログ」などの一時的な入力作業は、遷移元の画面状態を破棄しないよう、独立したダイアログ/ドロワーとして作業画面の上に重ねて表示します。これにより、元の画面に戻った際のデータの紛失リスクを排除しています。
