# ダッシュボード設計メモ

## 概要

このドキュメントは、ミーティング駆動型施設運営システムのダッシュボード設計に関するテストIDとルーティングの対応、および設計思想をまとめています。

## ダッシュボードの構成

### タブ構造と役割

| タブ名 | 役割 | 対象ユーザー | 利用タイミング |
|--------|------|--------------|----------------|
| 🏠 今日の全体状況 | ふだん見る"センターハブ" | 全職員 | 日中業務 |
| ☀ 朝のミーティング | 9:00時点の"スタートボタン" | 朝会参加者 | 9:00頃 |
| 🌙 夕方のミーティング | 17:15時点の"締めボタン" | 夕会参加者 | 17:15頃 |
| 🏥 医療・健康 | 看護担当・医療的ケアの"参照専用ビュー" | 看護師・医療担当者 | 随時 |
| 📋 支援記録 | 支援者が"個別の掘り下げ"をする入口 | 支援職員 | 個別支援時 |
| 📈 詳細分析 | 管理・改善チームの"振り返り/分析"専用ルーム | 管理者・改善担当者 | 月次・四半期分析 |

## ルーティング構造

### 基本ルート

- `/` (index) → DashboardPage (デフォルトで今日の全体状況タブ)
- タブ切り替えは内部状態管理（URL変更なし）

### クエリパラメータ

- `?mode=morning` → 朝のミーティングタブを開く
- `?mode=evening` → 夕方のミーティングタブを開く

## E2Eテスト観点

### クリティカルパス

1. **日常業務フロー**
   - 🏠 今日の全体状況での情報確認
   - Safety HUDでの安全状況把握
   - 記録進捗の確認

2. **朝会フロー**
   - ☀ 朝のミーティングタブへの切り替え
   - 安全確認チェックリストの実行
   - 優先フォロー利用者の確認

3. **夕会フロー**
   - 🌙 夕方のミーティングタブへの切り替え
   - 記録完了状況の確認
   - 健康状態サマリの確認

### テスト推奨ポイント

- タブ切り替えの応答性
- Safety HUDの数値更新
- チェックリスト項目のクリック反応
- 予定重なり検出とガイド表示
- レスポンシブデザインの動作

## 設計思想

### 情報の役割分離

- **日中業務** と **会議** と **分析** が混ざらない
- 画面の"目的"が直感で伝わる
- 「今はどこを開けばいいのか」が一瞬で分かる

### ユーザー体験の最適化

- 新人さんがメニュー名を見ただけで用件が分かる
- 情報過多を避け、必要な情報を必要なタイミングで提供
- 長期運用に耐える直感的な設計
