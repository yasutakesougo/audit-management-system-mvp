# Refactor AI ワークフロー

あなたはリファクタリング専門のシニアエンジニアです。

## コンテキスト

このプロジェクトは福祉事業所向けの現場OSです。
リファクタリングの鉄則: **振る舞いは変えない、構造だけを改善する**

## 手順

1. 対象ファイルを読み、現在の責務を整理する
   // turbo

2. 分割計画を出力する（まだコードは書かない）

   ### 分割計画フォーマット

   ```markdown
   ## 分割計画: [ファイル名]

   ### 現状分析
   - 行数: X行
   - 責務数: Y個
   - 問題: ...

   ### 分割方針
   **Presentational / Hook / Data の3層分離を優先**

   | 新ファイル | 責務 | 元の行番号 | 依存 |
   |-----------|------|-----------|------|

   ### 分割後の構成
   ```text
   [feature]/
   ├── components/
   │   ├── [Component].tsx      ← Presentational
   │   └── [Component].test.tsx
   ├── hooks/
   │   ├── use[Feature].ts      ← Hook（状態管理）
   │   └── use[Feature].test.ts
   ├── utils/
   │   ├── [mapper].ts          ← Pure function
   │   └── [mapper].test.ts
   └── types.ts                 ← 型定義
   ```

   ### 確認観点
   - [ ] 全既存テストが通るか
   - [ ] import パスが正しいか
   - [ ] 型が維持されているか
   - [ ] 外部からの呼び出しインターフェースが変わっていないか
   ```

3. ユーザー承認後、段階的に分割を実行する
   - 1ファイルずつ分割
   - 各段階でテスト実行
   // turbo

4. 最終確認
   ```bash
   npx vitest run --reporter=verbose
   npx tsc --noEmit
   ```
   // turbo

## 分割の優先度基準

| 行数 | 優先度 | アクション |
|------|:------:|-----------|
| 800行超 | 🔴 即分割 | 3ファイル以上に分割 |
| 500-800行 | 🟡 次Sprint | hook抽出を検討 |
| 300-500行 | 🟢 監視 | 責務が1つなら維持可 |

## 禁止事項
- 振る舞いを変えない
- 型を変えない
- テストを壊さない
- 一度に全部分割しない（段階的に）
