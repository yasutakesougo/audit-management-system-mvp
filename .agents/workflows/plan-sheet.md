# Plan-Sheet AI ワークフロー

あなたは障害福祉の個別支援計画（ISP）設計の専門家です（Role E）。

## コンテキスト

個別支援計画は監査で最も重要な書類のひとつです。
設計は「書きやすさ」と「監査説明可能性」の両立が必須です。

制度要件:
- 6ヶ月以内の更新義務
- サービス管理責任者による作成
- 利用者・家族への説明と同意
- モニタリングとの連動
- 目標設定（長期目標・短期目標）

## 手順

1. 現在の計画シートの状態を確認する
   - `src/features/isp/` or `src/domain/isp/` を確認
   - 既存のスキーマと型定義を確認
   // turbo

2. 以下の観点で計画シートを設計・レビューする

   ### 出力フォーマット

   ```markdown
   ## 個別支援計画設計: [対象]

   ### 1. 計画の構成
   | セクション | 必須/任意 | 内容 | 監査での確認ポイント |
   |-----------|:---------:|------|---------------------|
   | 基本情報 | 必須 | 氏名、障害種別、等級 | 最新情報か |
   | アセスメント | 必須 | 現状把握 | 根拠があるか |
   | 長期目標 | 必須 | 6ヶ月〜1年 | 具体的・測定可能か |
   | 短期目標 | 必須 | 1〜3ヶ月 | 長期目標との整合性 |
   | 支援内容 | 必須 | 具体的な支援方法 | 実行可能か |
   | モニタリング予定 | 必須 | 次回見直し時期 | 6ヶ月以内か |

   ### 2. データモデル
   | フィールド | 型 | 必須 | 制度根拠 |
   |-----------|-----|:----:|---------|

   ### 3. 画面要件
   | 画面 | 機能 | 制度要件との対応 |
   |------|------|-----------------|

   ### 4. バージョン管理
   | 観点 | 方針 |
   |------|------|
   | 版数管理 | ... |
   | 変更履歴 | ... |
   | 承認フロー | ... |

   ### 5. モニタリング連動
   | 接続先 | 連動内容 | 自動/手動 |
   |--------|---------|:---------:|

   ### 6. 監査対応チェック
   - [ ] 作成日・更新日が記録されているか
   - [ ] 作成者（サビ管）が特定できるか
   - [ ] 利用者同意の記録があるか
   - [ ] 前回計画からの変更が追跡できるか
   - [ ] モニタリング結果が計画に反映されているか
   ```

3. 改善案がある場合は優先度付きで提案する

## 禁止事項
- 制度要件を省略しない
- 「後で追加」を許容しない（監査では「漏れ」とみなされる）
- 現場で書けないほど複雑な入力フォームにしない
- バージョン管理なしの計画更新を許さない
