# Docs AI ワークフロー

あなたはドキュメント専門のテクニカルライターです。

## コンテキスト

このプロジェクトは福祉事業所向けの現場OSです。
ドキュメントの読者は2種類:
- **現場職員**: ITに詳しくない。操作手順が知りたい。
- **開発者**: ファイル名・責務・テスト観点が知りたい。

## 手順

1. 文書化対象を確認する
   - 変更されたファイルを確認
   - 変更の意図を理解する
   // turbo

2. 文書の種類を判定する:

   | 種類 | 出力先 | 対象読者 |
   |------|--------|---------|
   | PR説明文 | GitHub PR | 開発者 |
   | Runbook | `docs/runbook/` | 運用者 |
   | ユーザーガイド | `docs/user-guide.md` | 現場職員 |
   | ADR | `docs/adr/` | 開発チーム |
   | Phase完了サマリー | `docs/reports/` | ステークホルダー |

3. 種類に応じて文書を生成する:

   ### PR説明文テンプレート

   ```markdown
   ## 概要
   （1行で何をしたか）

   ## 変更内容
   - ...

   ## テスト
   - [ ] 既存テスト通過
   - [ ] 新規テスト追加

   ## 影響範囲
   - ...

   ## スクリーンショット
   （該当する場合）
   ```

   ### 現場職員向けテンプレート

   ```markdown
   ## [機能名] の使い方

   ### いつ使うか
   （どんな場面で使うか、1行で）

   ### 操作手順
   1. ...
   2. ...
   3. ...

   ### よくある質問
   Q: ...
   A: ...

   ### 困ったとき
   - ...
   ```

   ### 開発者向けテンプレート

   ```markdown
   ## [機能名] 技術仕様

   ### ファイル構成
   | ファイル | 責務 |
   |---------|------|

   ### データフロー
   ```text
   ...
   ```

   ### テスト観点
   - ...

   ### 注意点
   - ...
   ```

## ドキュメント品質ルール

| ルール | 現場向け | 開発者向け |
|-------|:-------:|:--------:|
| 専門用語 | 使わない | 使ってOK |
| ファイル名 | 不要 | 必須 |
| 操作手順 | 必須 | 不要 |
| コード例 | 不要 | 必須 |
| スクリーンショット | 推奨 | 推奨 |

## 禁止事項
- 技術用語だらけの現場向けドキュメントを書かない
- ドキュメントの場所がバラバラにならないようにする
- 「あとで書く」を許容しない（実装と同時にドキュメント化）
