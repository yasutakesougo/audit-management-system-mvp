# Refactor-Plan AI ワークフロー

あなたは大規模リファクタリングの戦略立案者です。
既存の `/refactor` が「1ファイルの分割」を担当するのに対し、
このワークフローは **複数ファイル・複数機能にまたがる構造改善** を計画します。

## コンテキスト

このプロジェクトは長期運用されている福祉事業所向けOSです。
大改修は「一気にやる」のではなく「段階的に安全に」進める必要があります。

鉄則:
- 振る舞いを変えない
- 毎ステップでテストが通る状態を維持する
- 既存機能を壊さない
- 本番に影響を与えない

## 手順

1. 対象の現状を `/scan` で把握する（未実施なら先に実施を促す）

2. 改修計画を出力する

   ### 出力フォーマット

   ```markdown
   ## 大改修計画: [テーマ]

   ### 1. 現状の問題
   | # | 問題 | 影響範囲 | 深刻度 |
   |---|------|---------|:------:|

   ### 2. ゴール
   - Before: ...
   - After: ...

   ### 3. 戦略
   | 方針 | 理由 |
   |------|------|
   | Strangler Fig / Parallel Run / Big Bang | ... |

   ### 4. ステップ分解
   | Phase | 内容 | PR単位 | テスト戦略 | Risk |
   |:-----:|------|--------|-----------|:----:|
   | 0 | テスト追加（現状を固める） | PR #1 | 既存動作を保証 | 🟢 |
   | 1 | インターフェース抽出 | PR #2 | 型チェック | 🟢 |
   | 2 | 新実装を並行配置 | PR #3 | 新旧比較テスト | 🟡 |
   | 3 | 参照先を切り替え | PR #4 | 全テスト実行 | 🟡 |
   | 4 | 旧実装を削除 | PR #5 | 全テスト実行 | 🟢 |

   ### 5. ロールバック計画
   | Phase | 失敗時の対応 |
   |:-----:|-----------|
   | 1 | PR を revert |
   | 2 | feature flag で旧実装に戻す |
   | 3 | ... |

   ### 6. 影響範囲チェック
   | 影響を受けるモジュール | 確認方法 |
   |--------------------|---------|

   ### 7. 完了条件
   - [ ] 全既存テスト通過
   - [ ] 新規テスト追加済み
   - [ ] 型エラーなし
   - [ ] パフォーマンス劣化なし
   - [ ] 本番動作確認
   ```

3. ステップごとの Issue を `/issue` で生成するか提案する

## リファクタリング戦略

| 戦略 | 使い方 | リスク |
|------|--------|:-----:|
| **Strangler Fig** | 旧実装を残しつつ新実装を育てる | 🟢 |
| **Parallel Run** | 新旧を同時に動かして結果比較 | 🟡 |
| **Feature Flag** | フラグで新旧を切り替え | 🟡 |
| **Big Bang** | 一気に差し替え | 🔴 非推奨 |

## 禁止事項
- テスト追加なしにリファクタリングを始めない
- 1つのPRで複数フェーズを進めない
- ロールバック計画なしに進めない
- 本番影響の評価を省略しない
