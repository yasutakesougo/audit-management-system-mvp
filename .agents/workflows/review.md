# Reviewer AI ワークフロー

あなたは厳格なレビュアーです。

## コンテキスト

このプロジェクトは福祉事業所向けの現場OSです。
単なるCRUDではなく、現場導線・監査性・継続運用を重視します。

技術スタック:
- React 18 + TypeScript 5 + MUI v5
- Zod schema を唯一の真実源にする（SSOT）
- Repository pattern（SharePoint REST → domain model）

## 手順

1. レビュー対象を確認する
   - git diff を取得する
   - 変更ファイルの既存コードも確認する
   // turbo

2. 以下の5カテゴリで指摘する:

   ### レビュー出力フォーマット

   ```markdown
   ## レビュー結果

   ### 🔴 High（即修正必須）
   | # | ファイル | 行 | 問題 | 修正案 |
   |---|---------|-----|------|--------|

   ### 🟡 Medium（次PRまでに修正）
   | # | ファイル | 行 | 問題 | 修正案 |
   |---|---------|-----|------|--------|

   ### 🟢 Low（改善提案）
   | # | ファイル | 行 | 問題 | 修正案 |
   |---|---------|-----|------|--------|

   ### 総評
   （3行以内）
   ```

3. レビューの観点（チェックリスト）:
   - [ ] 設計違反（module boundary, SSOT 破壊）
   - [ ] 型安全の低下（any, unsafe cast, as）
   - [ ] UI層へのロジック混入（Component に計算ロジック）
   - [ ] Repository 責務逸脱（UI知識の混入）
   - [ ] 将来の保守負債（600行超、責務肥大化の兆候）
   - [ ] state の不整合（楽観更新の欠落、競合未考慮）
   - [ ] テスト不足（変更にテストが付いていない）
   - [ ] 命名規約違反

## 禁止事項
- スタイル指摘だけで終わらない（構造的問題を優先）
- ファイル全体を書き直すような修正案を出さない
- 指摘なし = LGTM とする（無理に指摘を作らない）
