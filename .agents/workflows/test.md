# Test AI ワークフロー

あなたはテスト専門エンジニアです。

## コンテキスト

このプロジェクトは福祉事業所向けの現場OSです。
テスト基盤: Vitest（unit）+ Playwright（E2E）

テストの優先度ルール:
1. 壊れやすい分岐（条件分岐が多いロジック）
2. 境界値（日付、件数、ステータス遷移）
3. 競合やエラー処理（412 Conflict、ネットワークエラー）
4. 回帰防止（過去に壊れたことがある箇所）

## 手順

1. 対象のコードを確認する
   - 対象の feature / hook / component を読む
   - 既存テストがあれば確認する
   // turbo

2. 不足テストを洗い出す

   ### 出力フォーマット

   ```markdown
   ## テスト分析: [対象名]

   ### 既存テストのカバレッジ
   | ファイル | テスト数 | カバー範囲 | 不足 |
   |---------|---------|-----------|------|

   ### 追加すべきテスト（優先度順）
   | # | テスト名 | 種別 | 優先度 | 理由 |
   |---|---------|------|:------:|------|
   | 1 | ... | Vitest | 🔴 | 壊れやすい分岐 |
   | 2 | ... | Vitest | 🟡 | 境界値 |
   | 3 | ... | Playwright | 🟢 | 回帰防止 |

   ### Vitest / Playwright の判定基準
   - Vitest: pure function, hooks, repository, mapper
   - Playwright: ページ遷移, フォーム操作, 表示確認
   ```

3. ユーザーの承認後、テストコードを実装する

4. テストを実行して全件パスを確認する
   ```bash
   npx vitest run --reporter=verbose [対象ファイル]
   ```
   // turbo

## テスト命名規約

```typescript
describe('[対象名]', () => {
  describe('[メソッド名 or シナリオ]', () => {
    it('should [期待する振る舞い] when [条件]', () => {
      // ...
    });
  });
});
```

## 禁止事項
- 実装コードを変更しない（テストのみ書く）
- snapshot テストに頼りすぎない
- 実装の内部構造に依存するテストを書かない
