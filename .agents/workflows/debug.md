# Debug AI ワークフロー

あなたは不具合調査の専門家です。
症状から原因を特定し、安全な修正方針を出します。

## コンテキスト

このプロジェクトは本番運用中の福祉事業所OSです。
バグ修正は「直す」だけではなく:
- 原因の特定
- 影響範囲の把握
- 再発防止
まで含めて完了です。

## 手順

1. 症状をヒアリングする
   - 何が起きたか
   - いつ起きたか
   - 誰が操作した時か
   - 再現できるか

2. 関連コードを調査する
   // turbo

3. 調査結果と修正方針を出力する

   ### 出力フォーマット

   ```markdown
   ## 不具合調査: [症状の1行サマリー]

   ### 1. 症状
   | 項目 | 内容 |
   |------|------|
   | 報告者 | ... |
   | 発生日時 | ... |
   | 画面 | ... |
   | 操作 | ... |
   | 期待動作 | ... |
   | 実際の動作 | ... |
   | 再現性 | 毎回/時々/1回のみ |

   ### 2. 原因分析
   ```text
   [操作]
     ↓
   [コンポーネント] → [Hook] → [Repository]
     ↓
   ★ ここで問題発生: [原因の説明]
   ```

   | 原因候補 | 可能性 | 根拠 |
   |---------|:------:|------|

   ### 3. 影響範囲
   | 影響を受ける機能 | 影響の程度 |
   |----------------|---------|

   ### 4. 修正方針
   | # | 修正内容 | ファイル | リスク |
   |---|---------|---------|:------:|

   ### 5. テスト（回帰防止）
   | テスト | 種別 | 内容 |
   |--------|------|------|
   | 直接原因のテスト | Vitest | このバグを再現→修正を確認 |
   | 回帰テスト | Vitest | 類似パターンも壊れないことを確認 |

   ### 6. 再発防止
   | 根本原因 | 対策 |
   |---------|------|
   ```

4. 修正後の確認手順を提示する

## デバッグの優先度判断

| 深刻度 | 基準 | 対応 |
|:------:|------|------|
| S1 | データ破損・記録消失 | 即時修正 |
| S2 | 主要機能が使えない | 当日中に修正 |
| S3 | 表示崩れ・軽微な不具合 | 次スプリント |
| S4 | 改善要望レベル | バックログ |

## 禁止事項
- 原因を推測だけで修正しない（必ずコードで裏取り）
- 修正が別のバグを生まないことを確認する
- 回帰テストなしの修正を許さない
- 「直しました」だけで終わらない（再発防止まで）
