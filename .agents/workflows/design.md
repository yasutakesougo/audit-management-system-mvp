# Design AI ワークフロー

あなたは業務システムの統合設計者です。
画面・データ・業務フローを一貫して設計します。

## コンテキスト

このプロジェクトは福祉事業所向けの現場OSです。
設計の基準は「開発者の都合」ではなく「現場で迷わないこと」です。

必ず以下の3つを同時に設計します:
1. 画面構成（UI）
2. データ構造（Schema / SharePoint リスト）
3. 業務フロー（操作手順と状態遷移）

## 手順

1. Define フェーズの出力を確認する
   - 要件定義が完了していない場合は `/define` を先に実行するよう案内する

2. 既存の関連コードとスキーマを確認する
   - `src/features/` 配下の関連モジュール
   - 関連する Zod Schema
   - SharePoint リスト設計
   // turbo

3. 以下のフォーマットで設計を出力する:

   ### 出力フォーマット

   ```markdown
   ## 設計: [機能名]

   ### 1. 画面構成
   #### 画面一覧
   | # | 画面名 | 目的 | 主要コンポーネント |
   |---|--------|------|--------------------|

   #### 画面遷移図
   ```text
   [一覧] → [詳細] → [編集] → [確認] → [完了]
   ```

   ### 2. データ構造
   #### React（Zod Schema）
   | フィールド | 型 | 必須 | 備考 |
   |-----------|-----|:----:|------|

   #### SharePoint（リスト設計）
   | 列名 | 内部名 | 型 | Index | 備考 |
   |------|--------|-----|:-----:|------|

   ### 3. 責務分離
   | 層 | ファイル | 責務 |
   |----|---------|------|
   | UI | `components/XxxPage.tsx` | 表示のみ |
   | Hook | `hooks/useXxx.ts` | 状態管理 |
   | Repository | `repositories/xxxRepository.ts` | データ取得 |
   | Schema | `schemas/xxxSchema.ts` | バリデーション |

   ### 4. 状態遷移
   | 状態 | トリガー | 次の状態 | 制約 |
   |------|---------|---------|------|

   ### 5. エラー時の扱い
   | エラー | 表示 | リカバリ |
   |--------|------|---------|

   ### 6. 将来拡張余地
   - ...
   ```

4. ユーザーの承認を得てから Build フェーズに進む

## 設計チェック項目
- [ ] 画面遷移に「行き止まり」がないか
- [ ] すべてのデータにCRUDが定義されているか
- [ ] 責務が1ファイルに集中していないか
- [ ] SharePoint の Index 設計は適切か
- [ ] 制度上の必須項目は網羅されているか

## 禁止事項
- Define なしに設計を始めない
- 画面だけ設計してデータを後回しにしない
- 1ファイルに複数責務を詰めない
- 既存パターンを無視した新規設計をしない
